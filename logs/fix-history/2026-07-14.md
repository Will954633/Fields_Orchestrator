## [MONGO-ALLOWLIST-HARDEN] Refresh script updates firewall rule in place (no more delete+recreate gap) — 19:13 AEST
**Symptom:** Prophylactic. The allowlist refresh did `delete` then `create`; if `create` ever failed (billing lapse, transient API error, quota), the rule was left deleted and mongod firewalled off until a later successful run — this was the mechanism of the [WEBSITE-DOWN-BILLING-FW] outage.
**Root cause:** delete-then-create is non-atomic — there's a window where the rule doesn't exist, and a failed create makes that window open-ended.
**Fix:** When the rule exists, refresh now uses `gcloud compute firewall-rules update` (atomic, in-place) instead of delete+create. `create` is used only when the rule is genuinely absent (first run / self-heal). A failed apply now leaves the existing rule intact.
**Verification:** syntax-checked; perturbed live rule to 232 ranges → ran script → took UPDATE path → restored to exactly 231 (dummy gone, home IP intact); re-run = "No change" (idempotent); site API stayed HTTP 200 throughout (rule never dropped).
**Files:** `scripts/refresh-mongo-allowlist.sh`
**Recurrence:** First.
