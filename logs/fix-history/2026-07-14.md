# Fix History — 2026-07-14

## [WEBSITE-DOWN-BILLING-FW] Website data layer dead — closed GCP billing account deleted mongod firewall rule — 18:40 AEST (RESOLVED 19:03)
**Symptom:** Website returned HTTP 200 on all pages but was non-functional — data-driven pages/APIs failed. Netlify function `market-insights` returned `500 {"error":"...","message":"Socket 'connect' timed out after 10006ms"}`. All Netlify functions connect to mongod at `34.40.230.132:27017` (VM public IP) and could not reach it.

**Root cause (chain):**
1. GCP billing account `Fields Data Management 01` (`015552-D96B77-2718DA`) is **CLOSED** (`open: false`). Linked to project `fields-estate` but suspended — so `billingEnabled: true` at project level is misleading.
2. Nightly cron `refresh-mongo-allowlist.sh` (04:30 AEST) refreshes the `allow-mongodb` GCP firewall rule via delete+recreate. On **2026-07-11 04:30**, the delete succeeded but the create failed: `"This API method requires billing to be enabled."` — because billing was already closed. Result: firewall rule `allow-mongodb` (tcp:27017 from AWS us-east-1 EC2 ranges → tag `fields-orchestrator`) was gone.
3. Every subsequent run (07-12/13/14) died even earlier: `firewall-rules describe` on the now-missing rule errored under `set -euo pipefail`, so the script exited before it could recreate. **Script could not self-heal from a missing rule.**
4. With no firewall rule, all external TCP to 27017 is dropped → Netlify (AWS Lambda us-east-1) connect-timeout → website data layer dead.

**Fix:**
- **Code (done):** Made `refresh-mongo-allowlist.sh` self-healing — if `describe` finds no existing rule, treat current sourceRanges as empty so the diff registers a change and the rule is created fresh instead of the script aborting.
- **Infra (RESOLVED 19:03 AEST):** Will reopened the billing account (`open: true`). Ran `bash scripts/refresh-mongo-allowlist.sh` → recreated `allow-mongodb` (tcp:27017, 231 AWS us-east-1 EC2 ranges + home IP). Verified: `market-insights` API 200 w/ live data, `/api/v1/properties/for-sale` 200 (93 Robina), `/for-sale` page renders 222 listings (screenshot confirmed). Site fully restored.

**Wider risk:** A closed billing account puts the entire VM (mongod, orchestrator, all `fields-*` services, voice agent) at risk of being stopped by Google after the grace period. This is P0 beyond the website.

**Not the problem (ruled out):** DNS/hosting (200s OK), Netlify deploy (functions present; `/api/*` 404s in testing were wrong paths — real routes use `path:` config, e.g. `/api/market-narrative/<suburb>`), data freshness (target suburbs updated 2026-07-13, mongod healthy, 3013 active listings), mongod binding (0.0.0.0:27017, auth enabled), local firewalls (ufw inactive, iptables ACCEPT), VM IP (unchanged 34.40.230.132).

## [MONGO-ALLOWLIST-HARDEN] Refresh script updates firewall rule in place (no more delete+recreate gap) — 19:13 AEST
**Symptom:** Prophylactic. The allowlist refresh did `delete` then `create`; if `create` ever failed (billing lapse, transient API error, quota), the rule was left deleted and mongod firewalled off until a later successful run — this was the mechanism of the [WEBSITE-DOWN-BILLING-FW] outage.
**Root cause:** delete-then-create is non-atomic — there's a window where the rule doesn't exist, and a failed create makes that window open-ended.
**Fix:** When the rule exists, refresh now uses `gcloud compute firewall-rules update` (atomic, in-place) instead of delete+create. `create` is used only when the rule is genuinely absent (first run / self-heal). A failed apply now leaves the existing rule intact.
**Verification:** syntax-checked; perturbed live rule to 232 ranges → ran script → took UPDATE path → restored to exactly 231 (dummy gone, home IP intact); re-run = "No change" (idempotent); site API stayed HTTP 200 throughout (rule never dropped).
**Files:** `scripts/refresh-mongo-allowlist.sh`
**Recurrence:** First.
