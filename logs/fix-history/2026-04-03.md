## [TERMINAL-CRASH-LOOP] fields-terminal.service crash loop blocking worker agent — 08:17 AEST
**Symptom:** Worker agent produced no deliverables for 2026-04-03. Cron fired at 06:30 AEST but zero output in logs or deliverables directory.
**Root cause:** `fields-terminal.service` (node server.js on port 7681) was crash-looping every 5 seconds because `ttyd.service` (the actual working terminal) already held port 7681. The rapid restart cycle (~720 failed spawns/hour) caused resource contention at the exact moment the worker agent cron fired, killing the `claude` process before it could produce output.
**Fix:** Stopped and disabled `fields-terminal.service` — it's redundant since `ttyd.service` handles the same function. `sudo systemctl stop fields-terminal.service && sudo systemctl disable fields-terminal.service`.
**Files:** /etc/systemd/system/fields-terminal.service (disabled, not modified)

## [COVERAGE-FALSE-POSITIVE] Coverage check falsely flagging Robina/VL as critical — 09:55 AEST
**Symptom:** OPS dashboard showed Robina and Varsity Lakes as "critical" coverage gaps despite scraper running successfully. Scraper health showed last scrape as March 30 despite nightly runs.
**Root cause:** Two issues: (1) `coverage_check.py` compared Domain's search count (which includes recently-sold listings) against our `for_sale` count only — sold listings we correctly detected were counted as "missing". (2) `refresh-ops-context.py` read scraper freshness from `system_monitor.scraper_health` collection, but `write-scraper-health.py` was never triggered by the pipeline (phase2 completion condition never fires).
**Fix:** (1) Coverage check now adds `recent_sold` (last 14 days) to DB count before comparing with Domain — if `for_sale + recent_sold >= domain_count`, status is OK. (2) Scraper health in ops refresh now computes freshness directly from `last_updated` on Gold_Coast for-sale docs instead of relying on the broken `scraper_health` collection.
**Files:** scripts/coverage_check.py, scripts/refresh-ops-context.py, src/task_executor.py
**Recurrence:** First

## [AD-METRICS-DESYNC] Marketing monitor showing stale/blank ad data — 10:10 AEST
**Symptom:** Marketing monitor ad performance panel out of sync with Facebook. Active ads showing $0 spend, blank CTR/CPC/CPM. Paused ads with historical data showing blank last_30d and lifetime fields.
**Root cause:** Two issues: (1) `last_7d` was computed by summing locally-stored `ad_daily_metrics` rows, but new/recently-activated ads had no daily rows yet — the FB API aggregate was never queried for 7d per-ad data. (2) `last_30d` and `lifetime` from the FB Insights API omit long-paused ads (~12 of 72), and the `$set` upsert was overwriting existing DB data with empty `{}`.
**Fix:** (1) Added `fetch_ad_7d_insights()` to get authoritative per-ad 7d aggregates from FB API. `build_ad_profile` now uses API aggregate as primary source for `last_7d`, falling back to daily-computed. (2) `save_all` now skips writing `last_30d`/`lifetime` when the API returned None, preserving existing DB values from when the ad was active.
**Files:** scripts/fb-metrics-collector.py
**Recurrence:** First
