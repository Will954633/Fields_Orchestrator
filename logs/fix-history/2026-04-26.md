# Fix History — 2026-04-26 (AEST)

## [MIGRATION-CUTOVER-001] Cut Azure Cosmos → local MongoDB cutover for orchestrator services — 20:36 AEST
**Symptom:** Data was migrated from Azure Cosmos to local MongoDB on the VM, but the orchestrator daemon, trigger-poller, watchdog, valuation-api, and valuation-poller were all still connecting to Azure Cosmos. OPS_STATUS dashboard showed "Last run date: None" because the dashboard reads from local MongoDB but the daemon was still writing to Azure. Last successful pipeline run was 2026-04-22 (pre-migration); 3-day gap (Apr 23–25) followed.
**Root cause:** Three layers of stale Azure config overrode the new `.env` localhost URI:
1. `/etc/systemd/system/fields-orchestrator.service` had `Environment="COSMOS_CONNECTION_STRING=mongodb://...azure.com..."` hardcoded — systemd `Environment=` overrides values loaded from `.env` by python-dotenv.
2. `/etc/systemd/system/fields-claude-agent.service` had the same hardcoded Azure URI.
3. `config/settings.yaml` `mongodb.uri` still pointed at Azure Cosmos.
4. `src/orchestrator_daemon.py:107` had a defensive guard `raise ValueError("MongoDB URI must not be localhost in cloud deployment")` — this rejected the new localhost URI after the env was fixed.
**Fix:**
- Backed up the two systemd units to `*.bak.cosmos` and rewrote the `Environment="COSMOS_CONNECTION_STRING=..."` lines to point at `mongodb://fieldsadmin:***@localhost:27017/?authSource=admin`.
- Backed up `config/settings.yaml` to `settings.yaml.bak.cosmos` and updated `mongodb.uri` to the local URI.
- Removed the localhost-rejection guard in `src/orchestrator_daemon.py` (lines 100–107) — Azure has been cut off entirely, localhost is now the canonical destination.
- `systemctl daemon-reload` and restarted all 5 services. Confirmed via `/proc/$PID/environ` that the daemon now has `COSMOS_CONNECTION_STRING=mongodb://...@localhost:27017/...`.
**Files:**
- `/etc/systemd/system/fields-orchestrator.service` (URI changed)
- `/etc/systemd/system/fields-claude-agent.service` (URI changed)
- `config/settings.yaml` (URI changed; not pushed to GitHub per CLAUDE.md rule — contains credentials)
- `src/orchestrator_daemon.py` (removed cloud-deployment guard, lines 100–107)
**Recurrence:** First.

## [MIGRATION-CUTOVER-002] Fix 5 enrichment-script bugs surfaced by post-cutover manual run — 09:55 AEST (2026-04-27)
**Symptom:** Manual orchestrator run (started 21:44, completed 03:27) succeeded for core scraping/sold/valuation but failed 8 enrichment steps. Two distinct error patterns:
1. `pymongo.errors.ServerSelectionTimeoutError: SSL handshake failed: localhost:27017: [SSL: UNEXPECTED_EOF_WHILE_READING]` — killed step 12 (Enrich Property Timeline), step 14 (Suburb Statistics), and contributed to step 105 (Photo Analysis) fallbacks.
2. `❌ Failed to connect to MongoDB: name 'uri' is not defined` — appeared twice (step 102 first attempt, step 104 subprocess).
**Root cause:**
1. Four scripts had hardcoded `tls=True, tlsAllowInvalidCertificates=True` baked into `MongoClient(...)` from the Cosmos era. After Azure→local Mongo cutover, they tried to TLS-handshake against a non-TLS local Mongo and EOF'd.
2. `src/unknown_status_detector.py:81` had a typo: the conditional checked `"cosmos.azure.com" in uri` but the in-scope variable is `self.mongodb_uri`. Threw NameError every time the detector tried to connect.
**Fix:** Replaced unconditional `tls=True, tlsAllowInvalidCertificates=True` with conditional kwarg expansion `**({"tls": True, "tlsAllowInvalidCertificates": True} if "cosmos.azure.com" in <uri_var> else {})`. Forward/backward-compatible — TLS turns on automatically only for Cosmos URIs. Fixed the NameError by referencing `self.mongodb_uri`.
**Files:**
- `src/unknown_status_detector.py` (orchestrator repo, line 81 — NameError fix)
- `Feilds_Website/03_For_Sale_Coverage/enrich_property_timeline.py` (website repo, line 380)
- `Feilds_Website/03_For_Sale_Coverage/generate_suburb_statistics.py` (website repo, line 99)
- `Feilds_Website/03_For_Sale_Coverage/calculate_property_insights.py` (website repo, line 412 — preventative; not in the failed run but next call would have hit it)
- `Property_Data_Scraping/03_Gold_Coast/Gold_Coast_Wide_Currently_For_Sale_AND_Recently_Sold/Ollama_Property_Analysis/run_photo_reorder.py` (scraping repo, line 38)
**Recurrence:** First — but pattern (Cosmos-era hardcoded TLS) likely exists in other scripts. If new SSL handshake failures appear post-fix, grep for `tls=True\|ssl=True` across all repos and apply the same conditional pattern.
