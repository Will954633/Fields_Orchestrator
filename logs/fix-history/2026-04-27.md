## [SCRAPER-PRICE-PARSE] Domain price extractor returning empty string for "Offers Over" listings — 13:00 AEST

**Symptom:** 173 Ron Penhaligon Way (Robina) showed `price: "Auction"` in DB despite Domain having shown "OFFERS OVER $1,899,000" since 2026-04-12. Step 115 (track_price_changes) ran clean — it just never saw a new price to record. After re-scraping with fix, **86 listings across Robina/Burleigh Waters/Varsity Lakes had stale prices** (price tracker logged 86 changes in one run).
**Root cause:** Domain changed the DOM. The `<div data-testid="listing-details__summary-title">` wrapper now contains an empty `<span style="font-size:0">` spacer as its first child, then the actual price text in a nested `<div data-testid="listing-details__listing-summary-title-name">`. The parser at `html_parser.py:289` did `price_elem.find('span')` which grabbed the empty spacer span and stored `data['price'] = ''`. Then the merge logic at `parse_listing_html:64` (`if key not in property_data and value`) — the key WAS in property_data but the value was empty — fell through silently, and the `price` field was never set, leaving stale DB values untouched. The downstream save logic in `run_curlffi_suburb_scrape.py:642` filters new fields against `skip_fields`, but if `price` isn't in the parsed dict at all, the existing DB value stays.
**Fix:** Replaced the brittle `find('span')` with a lookup for the inner `data-testid="listing-details__listing-summary-title-name"` element, with fallback to the wrapper's full text. Added an `if price_text:` guard so only non-empty extractions are written. After the fix: re-scraped 138 active target listings; 86 prices were corrected; Ron Penhaligon went from `"Auction"` → `"OFFERS OVER $1,899,000"`, with `price_history` correctly appending the change event.
**Files:** `Property_Data_Scraping/07_Undetectable_method/00_Production_System/02_Individual_Property_Google_Search/html_parser.py` (lines 289-299). Pushed to `Will954633/Property_Data_Scraping` as commit `c7ae48b`.
**Recurrence:** First. Related class: prior Domain DOM changes broke scrapers in Feb/Mar 2026 (per backup files `*.backup_chrome_fix_20260212` etc.) — Domain's CSS-Module class names rotate but `data-testid` attributes are usually stable. Lesson: prefer the most-specific testid (`listing-details__listing-summary-title-name`) over generic descendant lookups (`find('span')`) — bare descendant lookups break the moment Domain inserts wrapper elements.
