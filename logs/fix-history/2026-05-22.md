## [TIER3-UNLOCK] Street View vision + features.basic fallback closes the cadastral-only narrative gap — 18:35 AEST

**Symptom:** Tier-3 properties (cadastral-only, never listed on Domain — e.g. 3 Lumley Street, Mudgeeraba) produced near-empty mini-sites. The full narrative chain (scarcity → positioning → personas → buyers) was gated on `scarcity_features` finding ≥1 notable feature, which itself reads `features.basic`. With no `property_valuation_data` and no `house_plan`, `features.basic` was empty → no notable features → entire chain skipped.

**Three pieces shipped:**

1. **`scripts/property_reports/inline_street_view.py` (new, ~290 lines)** — universal kerb-level baseline asset for any address with lat/lng. Fetches Google Static Street View (640×640, FOV 80, pitch 0) via the dedicated `GOOGLE_STREETVIEW_API_KEY` env var, uploads to blob storage, runs GPT-4o vision against a structured prompt that returns 5 category buckets (`dwelling`, `frontage`, `exterior`, `parking`, `street_context`) + a `narrative` block (`frontage_description`, `street_setting`, `buyer_appeal[]`, `visible_trade_offs[]`, `kerb_summary`). Persisted to `street_view_analysis` on the subject doc; cached so subsequent visits are free. Metadata-endpoint pre-check handles addresses Google has no coverage for (private roads, new estates).

2. **`scripts/property_reports/slot_resolver.py`** — new `street_view` resolver step between `satellite` and `gallery`. Also fixed an **in-memory subject mirror bug**: previously `resolve_satellite` and `resolve_street_view` wrote their analysis to the DB but didn't mutate `self._subject`, so downstream `derive_features_basic` read the stale in-memory doc and missed the visual signals on the same run. Now both write to `self._subject["satellite_analysis"]` / `["street_view_analysis"]` immediately after producing the analysis. Without this fix, the Tier-3 unlock would only work on the *second* re-resolve.

3. **`scripts/property_reports/inline_features.py`** — `derive_features_basic` now falls back to Street View + Satellite signals when the scrape-derived fields are missing:
    - `number_of_stories` ← `street_view_analysis.categories.dwelling.storeys` (1, 2, or 3)
    - `car_spaces` ← `parking.garage` ("double_attached"→2, "single_attached"→1, "triple_attached"→3, "carport"→1, "none_visible"→0)
    - `renovation_level` ← `dwelling.condition_impression` (recently_renovated→5 … needs_major_work→1)
    - `cladding_level` ← `exterior.primary_cladding` (weatherboard/fibro→1, brick→2, rendered→3, stone→4)
    - `bedrooms` ← `dwelling.estimated_bedrooms` parsed via `re.match(r"(\d+)", ...)` from "3-bed" / "4-bed" / "5+ bed"
    - `pool_present` / `water_views` ← already feeding from `satellite_analysis.categories.amenity_premiums` (pre-existing wire)

**Frontend (commits earlier in the day):**

- `street_view` added to canonical 15-step `STEP_ORDER` (build progress + frontend BuildingPage).
- `PropertyReportDoc.property.street_view` typed end-to-end through to `ReportFixture.streetView`.

**Operational fix:** ANTHROPIC_API_KEY had silently disappeared from `.env` between May 20 and today (still in `.env.bak.20260520-202545`). All narrative resolvers (scarcity / positioning / personas / buyers) had been logging "ANTHROPIC_API_KEY not set — skipping" and silently producing nothing — caught when 3 Lumley re-resolve generated visual data but skipped narratives. Restored from backup. **Other resolver work between May 20 and 22 may also have silently skipped narratives** — worth a one-off audit pass on docs touched in that window.

## [SAT-ANNO-003] Drop GPT bbox feature detection — keep only boundary + pin — 17:30 AEST

**Symptom:** Even with the cadastral boundary + point-in-polygon filter (SAT-ANNO-002), GPT-4o's bbox spatial accuracy on satellite tiles was unreliable enough that boxes regularly landed on the wrong pixel region (a tree got labeled "Pool", a patch of grass got labeled "Driveway"). The filter dropped those, leaving inconsistent presentation: sometimes 3 useful boxes, sometimes 0, with no predictability.

**Fix:** Stripped `detect_features()` + `filter_features_against_boundary()` + numbered-badge drawing from `satellite_annotation.py`. The annotated image is now just the satellite tile + bright yellow cadastral polygon + copper Fields drop pin. No rectangles, no badges, no legend.

**The structured GPT analysis (categories + narrative) was preserved** but now receives the **boundary-marked image** with an updated `USER_PROMPT_BOUNDARY` variant in `step117_satellite_analysis.py`. The new prompt explicitly tells GPT: *"The subject property's lot boundary is marked with a BRIGHT YELLOW POLYGON drawn directly on the image. For every 'categories' field: describe ONLY what falls INSIDE the yellow polygon. A pool on a neighbouring lot does NOT count as 'pool_visible: true'."* `analyse_satellite_image()` now accepts a `boundary_drawn: bool` parameter and switches prompts accordingly.

**Inline resolver flow reordered**: fetch tile → draw boundary on tile → send boundary-marked tile to GPT analysis → draw drop pin → save. Both Path A (existing-doc upgrade) and Path B (fresh) now re-run analysis against the boundary-marked image, so legacy docs get their categories cleaned up too.

**Staleness signal extended:** `_has_annotated_image()` returns False whenever the cached annotation has a non-empty `features` array (i.e. came from the box-bounding era). Every property hit refreshes once.

**Frontend** dropped the numbered feature legend (`AerialFeatureLegend` removed from `YourHomeTab`). Aerial view section is now image + findings card only.

**Cost change:** Saves ~$0.05 per fresh build (one GPT-4o call removed). Net per-build cost ~$0.37-0.42 again before TIER3-UNLOCK added Street View back at +$0.05.

**Recurrence:** Third major iteration on satellite annotation (SAT-ANNO-001 boundary polygon + filter; SAT-ANNO-002 stale-annotation upgrade + PIP filter; SAT-ANNO-003 drop GPT bbox entirely). The lesson: **GPT-4o is great at identifying what's in an image, unreliable at where**. Use it for classification, not coordinates. The yellow cadastral polygon (from authoritative GCCC data) is the visible proof of analysis tech.

**Files:**
- `scripts/property_reports/satellite_annotation.py` (GPT bbox detection removed; only `draw_lot_boundary` + `draw_annotations` for pin retained)
- `scripts/property_reports/inline_satellite.py` (Path A re-analyses against boundary-marked image; `boundary_marked_bytes` threaded through)

## [SAT-ANNO-002] Satellite annotation: stale annotations + bad amenity bboxes (pool on tree, driveway on grass) — 16:40 AEST

**Symptom:** Will reported the aerial view section for 36 Huntingdale Crescent showed (a) a "Pool" box drawn over a tree, (b) a "Driveway" box drawn on grass — neither in the correct location, and (c) no visible yellow lot boundary even though the cadastral boundary code shipped 2026-05-20.

**Root causes:**
1. **Stale annotations.** `_has_annotated_image()` returned True whenever `annotated_image_url` was set, so docs annotated BEFORE the 2026-05-20 boundary feature shipped were treated as current. The upgrade path never re-ran for them. 36 Huntingdale's annotation was generated against the old code path (no boundary fetch, no PIP filter), and the live page kept serving it.
2. **GPT-4o spatial accuracy.** Even with the boundary code, the prompt that says "only annotate amenities inside the yellow polygon" doesn't perfectly constrain bbox placement — the model frequently identifies real features but draws the box on the wrong pixel region. For 36 Huntingdale's old annotation: a tree got labeled "Pool", and a patch of grass got labeled "Driveway".

**Fix:**
1. **Staleness signal via `boundary_polygon`.** New persisted field on `satellite_analysis`. `_has_annotated_image()` now returns False when `annotated_image_url` exists without `boundary_polygon` — forces a fresh annotation pass for any doc generated pre-boundary-code. Roughly every property annotated before today triggers a one-time upgrade (~$0.05 each), then is stable.
2. **Point-in-polygon filter.** New `filter_features_against_boundary()` in `satellite_annotation.py`. For every `amenity` feature, samples a 5×5 grid of points inside the bbox and computes overlap with the cadastral lot polygon. Drops the feature if overlap < 30% of bbox area. `subject` / `detractant` / `context` features are kept as-is.
3. **Threshold rationale.** 30% catches mid-edge cases like driveways that legitimately straddle the lot/road boundary, while rejecting pool-on-neighbour and tree-mislabelled-as-pool (both score ~0%). Both filtered features for 36 Huntingdale logged at 0% overlap.

**Verification on 36 Huntingdale:**
- Force-resolve logs: `satellite: existing analysis without annotation — upgrading` → `got boundary polygon (7 vertices)` → `dropped 'Outdoor Entertaining Area' (amenity) — only 0% of bbox inside lot boundary` → `dropped 'Driveway' (amenity) — only 0% of bbox inside lot boundary`.
- Live page now shows the bright yellow lot boundary polygon, the copper Fields drop pin on the subject home, the Subject Property box, and a clean legend (none of the misidentified amenities).

**Files:**
- `scripts/property_reports/satellite_annotation.py` (PIP filter + `boundary_polygon` in return shape, pushed)
- `scripts/property_reports/inline_satellite.py` (staleness signal + persist `boundary_polygon`, pushed)

**Cost:** One-off $0.05 per pre-boundary doc on first re-resolve. Subsequent visits free.

**Recurrence:** Second instance of "GPT amenity wrongly bbox'd" — first was 12 Longueville Court's neighbour-pool on 2026-05-20, which prompted the boundary system. That one was solved by the polygon + prompt; this one needed the programmatic post-filter because the prompt-only approach still leaks past GPT's spatial drift.

**What's still imperfect:**
- Features GPT correctly identifies but mis-positions (e.g. driveway IS on the subject lot but the bbox got drawn 100 px south) will also be filtered. Better-than-wrong-data, but worse than perfect annotation. A future iteration could send the cropped boundary-only patch to GPT for a tighter bbox attempt before falling back to filter.

## [COVERAGE-AUDIT-008] Step 2 rescrape: Domain profile pages for never-imaged houses — 17:14 AEST

**Symptom:** After COVERAGE-AUDIT-007, Tier-1 Analyse-Your-Home coverage was still 14.8% Robina / 12.7% BW / 9.9% VL. Big remaining gap was houses with no images at all on the cadastral docs (Robina 892, BW 376, VL 397 — units excluded since Domain profile pages are house-only).

**Hypothesis:** Domain.com.au has a `property-profile/<slug>` page for nearly every address in their dataset, regardless of whether it has ever been listed. Hitting these pages would yield bed/bath/parking/lot/valuation/comparable-sales data — possibly photos and floor plans if Domain has held them. Coverage gain would lift multiple Tier gates at once, not just floor plans.

**Fix:** Ran `scripts/scrape_property_profiles.py --suburbs robina,burleigh_waters,varsity_lakes --only-missing-images --all` (existing tool, no code changes needed). 30 workers, 2s rate, Bright Data Web Unlocker. Queue size 1,202 after filtering (units, recent-scrape, already-failed records excluded).

**Results — disappointing:**
- Attempted: 1,202
- Profile pages fetched successfully: 1,035 (86%)
- Successfully parsed: 117 (10% of attempted)
- Failed parse: 918 (76% — Domain returned a page but `__APOLLO_STATE__` had no listing data)
- Failed fetch: 167 (14% — Bright Data couldn't get a usable response)

The 86%-fetch-but-10%-parse pattern means Domain has profile-page **shells** for these addresses but **no embedded listing data**. Properties that have never been on Domain's marketplace don't get the rich JSON payload, even if Domain's address index knows the address exists.

**Of the 117 successfully-parsed records:**
