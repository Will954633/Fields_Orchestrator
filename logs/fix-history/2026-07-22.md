## [SITEMAP-PUSH-SILENT-FAILURE] Sitemap.xml frozen at 2026-07-20 snapshot for 2 days — Google indexing new property pages against stale data — 08:20 AEST
**Symptom:** Will reported a property page published "yesterday" (14 Elfin Street, Robina) wasn't appearing anywhere in Google search results, while other pages had indexed much faster. Asked me to find the problem, believing it was in the sitemap/exposure mechanism.
**Investigation:** Confirmed the page itself was fine — live, 200, SSR-rendering full content, valid `<link rel="canonical">`, no noindex, present in the live sitemap.xml, robots.txt open. GSC URL Inspection showed `coverageState: "Discovered – currently not indexed"`, `lastCrawlTime: null` — Google knew the URL existed but had never fetched it. Sampled 5 more properties from the same batch: 3 already indexed (crawled 2026-07-16/07-20, i.e. BEFORE their editorial was even published — the underlying page was already known), 2 came back "URL is unknown to Google" entirely.
**Root cause (two compounding factors):**
  1. **Silently broken push pipeline.** `system_monitor.seo_indexation_snapshots` history showed sitemap URL count 170 (07-16) → 1,690 (07-19) → 18,686 (07-21) — an 11x jump exactly coinciding with the off-market paid report system going live 2026-07-20 (adds `/off-market/<slug>` URLs, deliberately `index, follow`). This pushed `public/sitemap.xml` to ~3.9MB. The VM's daily cron (`regenerate-sitemap.sh`, 06:15 AEST) regenerates the file correctly every day but its `gh api ... --field content=$(base64 ...)` push step has been failing EVERY DAY since 2026-07-20 with `Argument list too long` (base64 payload as a CLI arg blew past the 2MB `ARG_MAX`) — this is the exact documented `gh_api_large_file_gotcha` class of bug, just not applied to this particular script. Because the failing command was the first operand of a `cmd && echo ...` list, bash's `set -e` didn't abort the script, so it printed "Done." and exited 0 every day — completely silent. Last successful push was `e87535b` at 2026-07-20T08:05:17Z. Every property published/updated/scraped/status-changed since then (the entire 76-property backlog batch, 14 Elfin Street's own editorial, likely under_contract/withdrawn transitions) was invisible to whatever Google fetched from the live sitemap.
  2. **Crawl-budget dilution (secondary, not fixed — flagging for a decision).** The 16,866 new `/off-market/` URLs now outnumber the 1,731 actual `/property/` pages ~10:1 in the same public sitemap. This reproduces the exact "low domain authority + high URL count → Google won't spend crawl budget" constraint already documented as this site's core SEO bottleneck since March 2026 (see `[[seo_indexation_baseline]]`) — just via a different page type than the sold-page dump that caused it originally. Worth a deliberate decision on whether `/off-market/` should get the same `noindex` treatment already applied to the analogous `/your-home/` mini-site path — did not make this call unilaterally since it's a product/SEO policy tradeoff, not an unambiguous bug.
**Fix:** Rewrote the push step in `regenerate-sitemap.sh` to build the PUT body as JSON on disk and pass via `--input` (the established large-file pattern) instead of `--field content=`. Also replaced the "skip if unchanged" check — which compared against the Contents API's `.content` field, silently empty for any file >~1MB, a second latent bug — with a git blob-SHA comparison against the already-fetched `.sha`, which works at any file size. Verified live: ran the script manually, it generated 19,224 URLs and pushed successfully (commit `4425a3b`) where it had failed 100% of the time since 07-20. Waited for Netlify to redeploy and confirmed the live `sitemap.xml` now shows `lastmod=2026-07-21` for 14-elfin-street-robina (previously stuck at `2026-07-19`). Resubmitted to GSC.
**Bonus fix (surfaced during this investigation, unrelated to sitemap):** last night's orchestrator run (07-21 23:20 AEST) had step 112 (property type classification) crash with `TypeError: 'NoneType' object is not iterable` in `shared/claude_vision.py:151` (`resp.content` was `None` on some OpenRouter response, and the code assumed it was always an iterable list) — this bug existed since the vision-OpenRouter migration but was masked because every call was previously failing on the dead direct-API-key 400 first. My 2026-07-21 OpenRouter routing fix made calls succeed, which is what newly exposed this. Fixed by guarding `for b in (resp.content or [])`. Verified live: reprocessed 15 Robina properties through step 112, 0 errors (previously crashed on this exact class of response). This is why the watchdog escalation below fired.
**Files:** `01_Website/scripts/regenerate-sitemap.sh` (commit `4425a3b`, `Website_Version_Feb_2026`), `shared/claude_vision.py` (commit `6777be1`, `Fields_Orchestrator`)
**Recurrence:** Nth instance of the `gh api` large-file class of bug (see `[[gh_api_large_file_gotcha]]` memory) — this specific script just hadn't been touched since the sitemap grew past the threshold. Nth instance of "dead direct-API-key masked a real bug underneath" (see 2026-07-21 `[NIGHTLY-EDITORIAL-SATELLITE-CASCADE]`) — underscores that fixing a masking failure can immediately surface a second, previously-invisible one; always re-verify the newly-unblocked path actually completes cleanly, not just that the original error is gone.
