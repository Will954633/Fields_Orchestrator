## [COVERAGE-AUDIT-002] Robina backfill test batch — root cause of v2-scrape failures is wrong cadastral postcodes — 15:55 AEST

**Symptom:** 4,981 Robina cadastral records show `scraped_v2_failed_at`. Unknown whether this is a Bright Data / Akamai problem (retryable), a Domain-side absence (no profile page), or a URL-construction bug.

| Status | Count | Note |
|---|---:|---|
| SUCCESS | 2 | Both had `parsed_image_count=0` — Domain has the profile page but no listing photos |
| PARSE_FAIL | 8 | All returned Domain's `__NEXT_DATA__` page `/_error` with `statusCode: 404` — the slugged URL doesn't exist on Domain |

**Root cause of the parse-fails:** Inspected the failing URLs by manually retrying with `postcode=4226`:

| Original URL (from cadastral) | Result | Retried URL (4226) | Result |
|---|---|---|---|
| `703-glades-drive-robina-qld-4213` | 404 | `703-glades-drive-robina-qld-4226` | PROPERTY_EXISTS |

| `839-legend-trail-robina-qld-4213` | 404 | `839-legend-trail-robina-qld-4226` | PROPERTY_EXISTS |
| `38-st-ives-drive-robina-qld-4226` | 404 | (same) | 404 (genuine — slug not on Domain) |

The cadastral baseline has 3,268 Robina records (28% of 11,760) with non-4226 postcodes — split 4227:1317, 4218:896, 4213:599, 4220:456. These postcodes are adjacent suburb codes (Varsity Lakes, Mermaid Waters, Mudgeeraba, Burleigh) that the cadastral data assigns to addresses Domain considers part of Robina (golf course estates, suburb edges). **1,650 of these 3,268 wrong-postcode records are flagged `scraped_v2_failed_at`** — i.e. ~33% of the 4,981 v2-failures are recoverable by retrying with postcode 4226.

**Critical secondary finding:** Both SUCCESS cases returned `parsed_image_count=0`. Cadastral-only addresses (never listed) often have a Domain profile page but no listing photos — we can't backfill images for those from Domain. The image-coverage ceiling for cadastral-only addresses is set by satellite analysis, not listing scrapes.

**Fix proposed (not yet applied):**
1. Update `scripts/scrape_property_profiles.py:build_profile_url` to try `qld-4226` as a fallback when `LOCALITY=ROBINA` regardless of cadastral POSTCODE, OR
2. Normalise cadastral `POSTCODE=4226` wherever `LOCALITY=ROBINA` AND POSTCODE in {4213,4218,4220,4227} (one-shot data fix), then re-run v2 scrape with `--retry-failed --only-missing-images --suburbs robina`.
3. Mark `scraped_v2_failed_reason='domain_404'` instead of `'parse'` when we detect `/_error` status, so we stop confusing structural absence with retryable failure.

## [COVERAGE-AUDIT-004] Floor plans extracted from v2 image arrays via GPT-4o-mini vision — 18:08 AEST

**Symptom:** Floor-plan coverage on cadastral records was 393/11,953 = 3.3% in Robina, with the prevailing assumption that floor plans live only on listing pages (not Domain property-profile pages). Will pointed out the assumption was wrong: Domain's profile page for 16 Tullamarine Dr embeds the floor plan as image 29 of 29 in the `media({"categories":["IMAGE"]})` array, untagged. The data was sitting in the v2 scrape results we already had — we just weren't extracting it.

**Diagnosis:** Domain doesn't tag floor plans separately (`category: "property"` for all images). Listing agents upload them in arbitrary positions — most often near the end, but not reliably last. A 6-record visual sample showed 1/6 last images = floor plan; expanding to multi-listing aggregations (Legend Trail records with 100+ images), floor plans can be buried at position 121/161. Position heuristics fail; visual classification is required.

**Fix:** Built `scripts/extract_floor_plans_from_v2_images.py` — GPT-4o-mini vision classifier with structured "YES/NO architectural floor plan?" prompt at `detail=low`. Per-record URL dedup; global thread-safe URL cache to skip API calls on shared images across records (saves ~5-15% of calls on duplicate-heavy datasets like golf-estate cohorts). Output written to non-destructive `floor_plans_v2_extracted[]` field + `floor_plans_v2_classifier` metadata (model, prompt version, per-candidate verdicts). Idempotent (skip if `floor_plans_v2_extracted_at` set; `--force` to override). Concurrent (`--workers` for records, `--image-workers` per record).

**Test (20 records, `--tail 3`):** 4/20 records found ≥1 floor plan (20% recall), 5 images extracted. **Precision: 5/5 manually verified = 100%.** Established the classifier works; recall capped by the last-3 search window.

**Full run (`--all --all-images --workers 20 --image-workers 8`):**
- Eligible: 2,736 Robina records with ≥5 v2 images
- Runtime: 27.5 min (1,652s)
- Total images classified: 102,565 (5,762 cache hits = 5.6% saved)
- API calls: 96,803 gpt-4o-mini vision at `detail=low` ≈ ~$14.50 estimate
- Records with ≥1 floor plan found: **1,578 (58%)**
- Floor plan images extracted: **2,723**
- Errors: 0
- Precision (12 random samples spot-checked): **12/12 = 100%** — every sample was a genuine architectural floor plan, several included site plans too

**Coverage delta (Robina):**

| Source | Records with FP | % of total cadastral |
|---|---:|---:|
| floor_plans (legacy, mostly dead Azure URLs) | 393 | 3.3% |
| floor_plans_original (live Domain CDN) | 390 | 3.3% |
| **floor_plans_v2_extracted (NEW)** | **1,588** | **13.3%** |

**Key win — coverage for never-listed cadastral records:** 1,448 floor plans now exist for properties that have never been listed on Domain. Previously zero data for these.

## [PATH-A-VALUATION] Auto-run valuation engine for new appraisal pipeline subjects — 16:15 AEST
**Symptom:** Page 10 of every appraisal PDF rendered "Analyst review required" forever. Homeowner-submitted subjects (e.g. 12 Beaconsfield Drive, Tusmore Place, 1 Glenside Drive) have no `valuation_data` because they have never been on the for-sale list, so the nightly batch precompute_valuations.py skips them, so §03 receipts had no comp cards to render.

**Root cause:** Two issues stacked.
1. No pipeline step existed to compute valuation_data for an on-demand subject. The batch script's per-subject entry point `precompute_property_valuation()` was reachable but had no wrapper, no process_commands entry, no bridge integration.
2. Even after wiring the engine, §03 receipts came back empty. `data_pull.section_03_receipts` reads `valuation_data.comparables[]`, but the engine writes the actually-used sold comps to `valuation_data.recent_sales[]` (the `comparables[]` field is reserved for current listings used as comps, which is almost always empty for residential subjects). Schema mismatch between engine output and report consumer.

**Fix:**
1. Built `scripts/run_subject_valuation.py` — wrapper around `precompute_property_valuation()` that loads catchment-only sold data (~30s vs 131s full GC) and writes back to `Gold_Coast.<suburb>.<subject_id>.valuation_data`.
2. Added process 301 to `config/process_commands.yaml` invoking the wrapper with `--pipeline-id`.
3. Updated bridge sync (`property_reports_to_appraisal_pipeline_bridge.py`) and `analyse-lead.mjs` to queue 301 then 300 (1s offset, FIFO sequential) for any new pipeline record whose subject lacks `valuation_data`.
4. Updated `data_pull.section_03_receipts` to read from `recent_sales[]` ∪ `comparables[]` filtered to `included_in_valuation=True`, mapped engine field names (`adjustment_result.adjusted_price`, `original_sale_price`) to the card schema, capped adjustment rows at 7 + rolled remainder into a "smaller adjustments (net)" line, normalised address (strip "QLD 4220" tail).
5. Added §03 receipts compact-variant CSS overrides in `09_Appraisals/Version_Four/preview.html` (template uses inline styles, so selectors had to target the inline-styled `.comp-card` table cells with `!important`).

**Verification:** Test pipeline 6a0aaac3b65866386eac3b96 on subject 12 Beaconsfield Drive Burleigh Waters. After auto-301+300: range $1.88M–$2.40M (confidence: high), page 10 renders 2 comp cards (37 Beaconsfield $2.14M adjusted, 23 Kestrel $2.30M adjusted) each with 8 adjustment rows, footer aligned, zero overflow, zero hard breaches.
