## [MINISITE-HEALTH] Mini-site data dictionary + completeness/freshness checker — 08:46 AEST

**Symptom:** No systematic way to know whether the dynamic data on each house mini-site
(`property_reports`) is complete and fresh, or which fields are stale/missing per home.
**Root cause:** New capability, not a regression. Mini-site fields span two freshness tiers
(nightly-refreshed "living" data vs build-time slots that only refresh on rebuild) with
freshness signals scattered across the report doc and several upstream `precomputed_*`
collections — easy to have silently stale data.
**Fix:** (1) Documented every dynamic field, its data flow, updating process, freshness
signal, completeness rule and stale-after threshold in `MINISITE_DATA_DICTIONARY.md`.
(2) Built `scripts/minisite_health_check.py` — evaluates all 18 reports field-by-field,
emits OK/STALE/MISSING/PENDING-EXPECTED/ERROR/UNKNOWN/KNOWN-GAP per field, uses
expected-nightly-run-date logic so market/living data flags the moment it misses one run,
and writes per-run snapshots to `system_monitor.minisite_health_snapshots` for "last changed"
tracking. Two parsing bugs fixed during build (market_charts `_id` suburb prefix; valuation
freshness uses build-time `comps_resolved_at` not an unreliable subject lookup).
**Finding (real issue surfaced by the tool):** `activity_refreshed_at` is frozen at build
time fleet-wide (e.g. 25 Huntingdale stuck at 2026-05-23) while `data_pull_date` and
`comparables_refreshed_at` update nightly — the activity-feed refresh step is not running for
existing reports. Separately, several reports (1 Glenside, 9/12 Longueville, 3 Lumley) are
fully frozen since mid-May (stuck builds). To be investigated.
**Files:** MINISITE_DATA_DICTIONARY.md (new), scripts/minisite_health_check.py (new).

## [MINISITE-ACTIVITY] Home-tab activity feed frozen fleet-wide — 10:14 AEST

**Symptom:** `activity_refreshed_at` stuck at build time on every real report (e.g. 25
Huntingdale frozen at 2026-05-23) while `comparable_events`/`comparables_refreshed_at`
updated nightly — surfaced by minisite_health_check.
**Root cause:** `refresh_property_reports.py` nightly job refreshes the comparable feed for
every report, but only rebuilt the `activity[]` timeline (and stamped `activity_refreshed_at`)
for the hardcoded legacy Merrimac `REPORT_CONFIGS`. Real reports' activity feed was only ever
set once, at build time.
**Fix:** Added `rebuild_activity_feed()` — folds tonight's `comparable_events` (mapped to
activity items, newest-first) on top of the preserved build-narrative items, deduped and
capped, then sets `activity` + `activity_refreshed_at` inside `refresh_comparables_for_doc`
(runs for EVERY report). Verified on 25 Huntingdale: feed went from 16 stale build items to
10 items led by fresh "just listed" events; `activity_refreshed_at` now current. Takes effect
on the 23:30 AEST nightly cron automatically.
**Files:** scripts/refresh_property_reports.py.
**Recurrence:** First

## [SEARCH-INTENT] Search Intent Explorer button "not working" — slow/hanging load — 11:30 AEST

**Symptom:** The Search Intent Explorer button on the ops dashboard (Marketing tab) appeared
broken — the new tab hung/never painted. Will needed up-to-date market research.
**Root cause:** Not a crash and not stale data — the underlying collections are fresh
(latest 2026-06-04). `search-intent-explorer.mjs` loaded EVERY historical document from 11
collections (~50K rows) and embedded them inline as a single 14.4 MB HTML page. The page only
renders top-N per tab, so >95% of the payload was unused. Server build was 6.8s and the browser
needed ~11.5s to download+parse 14.4 MB — long enough that the tab looked dead and Will gave up.
Payload grew unbounded because the daily collectors keep appending.
**Fix:** Scoped the five fresh, high-volume collections (search_suggestions, search_paa_questions,
search_reddit_posts, search_trends, search_youtube_suggestions) to a recent window — default 30
days, overridable via `?days=N` (clamped 7..90) and shown as a "Last 30d" header chip. Left the
small/stalled-collector collections (ad_queries, search_console, youtube competitor channels/videos,
intent_analysis, scored_questions) loading in full so their tabs never go blank. Also truncated
Reddit `selftext` to 200 chars (it's only used for client-side search matching, never displayed —
saved >1 MB). Payload 14.47 MB → 6.20 MB live (will be ~5 MB once full 30d window is cached).
Verified headless: no JS errors, all 11 tabs populated (incl. the full-load Ads/GSC/YouTube tabs),
Overview renders fresh data.
**Files:** netlify/functions/search-intent-explorer.mjs (Website_Version_Feb_2026, commit 4cb7680).
