## [APPR-COVER-001] Appraisal cover hero rendered as a 150×100 grainy thumbnail — 14:05 AEST

**Symptom:** Will picked a cover hero photo for 12 Longueville Court via the ops dashboard `HeroImagePicker`. The re-rendered front cover showed an extremely grainy image. The chosen URL was `rimh2.domainstatic.com.au/6UvQn0eB5UpJrItAMRyrAr5TEBg=/filters:format(png):quality(80):no_upscale()/2018153941_1_1_221025_040937-w1024-h682` — filename implies 1024×682, but the signed hash actually resolved to a 150×100 PNG thumbnail (32,870 bytes). The on-disk `assets/img/cover_hero_<subject_id>.jpg` was identified by `file` as `PNG image data, 150 x 100`. Cover HTML CSS also referenced the signed URL directly, so the headless-chromium PDF render re-fetched the same thumbnail.
**Root cause:** Same class of bug as [FP-001](2026-05-20.md). The `rimh2.domainstatic.com.au` CDN signed-hash URL can silently return any resolution; only the `bucket-api.domain.com.au/v1/bucket/image/<filename>` endpoint reliably serves the original full-res file. The cover renderer in [generate_appraisal_v4.py](../../scripts/generate_appraisal_v4.py) wasn't using the canonical `shared.domain_urls.to_bucket_api_url()` helper (which already protects the floor-plan path).
**Fix:** Three call sites in `scripts/generate_appraisal_v4.py` now route Domain CDN URLs through `to_bucket_api_url()`:
1. `render.render_section_00_cover_html(... hero_image_src=to_bucket_api_url(...))` — the cover HTML now references the full-res bucket-api URL, so PDF renders pull the original file (confirmed via curl: signed → 32KB, bucket-api → 533KB).
2. The `cover_hero_image_src` download into `expected_hero` (`f"cover_hero_{subject_id}.jpg"`) — the local asset used by the next render is the full-res version.
3. The auto-fallback candidate list (`scraped_data_apr01_recovered`, `scraped_data`) — when no analyst pick exists, auto-selected photos are also bucket-api-normalized.

Also manually re-downloaded the on-disk hero for 12 Longueville (subject_id `690bd8af8b8f54659264d560`) using the helper, replacing the 150×100 PNG with the proper 1024×682 JPEG. No re-render needed for the asset itself, but the cached HTML still references the signed URL — next re-render via ops dashboard will pick up the new HTML.

**Files:** `scripts/generate_appraisal_v4.py` (3 edits + 1 import line); local file `artifacts/appraisals_v4/assets/img/cover_hero_690bd8af8b8f54659264d560.jpg` overwritten with full-res.
**Recurrence:** Second instance of the Domain signed-hash thumbnail trap (after FP-001). Worth a broader audit — any other code path that consumes Domain CDN URLs without going through `to_bucket_api_url()` is at risk. Candidates: AI editorial photo descriptions, Facebook Marketplace post images, possibly the YourHomePage hero overlay.
