## [FP-001] Floor plan GPT analysis failed — Domain CDN returned 150×107px thumbnail — 22:38 AEST

**Symptom:** 12 Longueville Court had no room dimensions extracted. `floor_plan_analysis.rooms` was empty and `parsed_rooms` null despite `floor_plans_v2_extracted` containing a URL that appeared to be a 3277×2338 floor plan image. Confirmed systemic: audit found 9,422 properties across 8 suburbs with the same risk — all using `rimh2.domainstatic.com.au` CDN URLs.
**Root cause:** Domain CDN (`rimh2.domainstatic.com.au`) encodes a signed resize hash in the URL. The hash can resolve to a thumbnail even when the filename suggests full resolution (e.g. hash `7T4qYJN7-Rx38686gyk_vspyyhw=` returned 150×107px for a `-w3277-h2338` filename). GPT-4o received an unreadable thumbnail. The bucket-api endpoint (`bucket-api.domain.com.au/v1/bucket/image/<filename>`) bypasses signing and always returns the original full-res file.
**Fix (comprehensive — works for ALL addresses, not just new listings):**
1. **Shared helper (`shared/domain_urls.py`)** — Canonical `to_bucket_api_url()` that converts any Domain CDN URL to bucket-api. Handles all observed variants: `/hash/filename`, `/hash/filters/filename`, `/hash/fit-in/dims/filename`, `/hash/fit-in/dims/filters/filename`, `/hash/filters/fit-in/dims/filename`. Regex uses `(?:/(?:filters:[^/]+|fit-in/[^/]+))*` to match transform segments in any order.
2. **On-demand resolver (`scripts/property_reports/inline_floor_plan.py`)** — Normalizes all URLs to bucket-api before sending to GPT, and prefers bucket-api in the URL ranker. Every address typed into "Analyse Your Home" (including off-market cadastral records, not just `for_sale`) is now safe.
3. **Scraper write-path (`extract_floor_plans_from_v2_images.py`)** — Uses shared helper at write time so all newly-extracted URLs are bucket-api by default.
4. **Reanalyze fallback (`reanalyze_floor_plans_gpt54.py`)** — Pre-converts URLs and falls back to bucket-api if downloaded image is below 500px shortest side.
5. **DB backfill** — Rewrote 9,418 CDN URLs across all 8 suburbs to bucket-api format. Initial backfill had a bug — first regex captured the filters segment as the filename for `fit-in` URLs (which order transforms differently). Repaired all 8,462 affected docs by recovering original URLs from the untouched `floor_plans_v2_classifier.candidates` field. Final state: 16,706 clean bucket-api URLs, 0 CDN, 0 malformed; spot-checks confirm all return real high-res images (600px+).
6. **Immediate data fix** — Re-ran GPT-4o on 12 Longueville Court → 12 rooms extracted, 268sqm internal area.
**Files:** `shared/domain_urls.py` (new), `scripts/property_reports/inline_floor_plan.py`, `scripts/extract_floor_plans_from_v2_images.py`, `scripts/reanalyze_floor_plans_gpt54.py`, DB backfill across robina/varsity_lakes/burleigh_waters/carrara/mudgeeraba/reedy_creek/merrimac/worongary.
**Recurrence:** First. Coverage: the on-demand resolver path means every address — for-sale, sold, off-market, cadastral-only — now gets a working floor plan on first view. Result is cached in `system_monitor.property_reports` so subsequent views are instant.

## [SAT-001] Satellite annotation incorrectly labels neighbour's pool as subject property amenity — 21:40 AEST

**Symptom:** 12 Longueville Court showed "Pool" in the satellite DETECTED ON THE IMAGE panel. Pool is on the neighbour's lot, not the subject property. satellite_analysis.categories.amenity_premiums.pool_visible was correctly false, and the narrative correctly noted "no pool on subject lot", but the features array contained a "Pool" entry with a bbox clearly outside the subject lot boundary.
**Root cause:** The satellite annotation prompt in `satellite_annotation.py` did not constrain amenity features to the subject lot. GPT annotated any visible pool in the image frame, including neighbours'. The features array bbox [690, 350, 750, 410] vs subject bbox [560, 580, 720, 740] — completely different area.
**Fix:**
1. **Cadastral boundary system** — new `scripts/property_reports/lot_boundary.py` fetches the exact lot polygon from GCCC ArcGIS Cadastre_Current_view service using point-in-polygon query (no API key needed).
2. **Boundary draw + crop** — `satellite_annotation.py` now: (a) fetches the boundary polygon, (b) projects ring vertices to pixel space using Mercator math, (c) draws a yellow boundary outline on the image, (d) crops to the lot bbox + 1.5× margin so the subject property fills more of the frame.
3. **GPT prompt update** — updated to reference the visible yellow polygon: "ONLY annotate amenity features that fall INSIDE the yellow boundary polygon."
4. **Immediate data fix** — removed the erroneous Pool feature from 12 Longueville Court's features array in DB. Also added no-boundary fallback prompt for properties where GCCC lookup fails.
**Files:** `scripts/property_reports/lot_boundary.py` (new), `scripts/property_reports/satellite_annotation.py`, `scripts/property_reports/inline_satellite.py`
**Recurrence:** First (but likely affects many properties with pools on neighbouring lots)

## [CA-009b] VL photo retry — throttled photo-level rate limiting — 20:34 AEST

**Symptom:** Varsity Lakes photo downloads triggered Domain CDN rate-limit repeatedly even with `--workers 8` (reduced from 24). ~2K photos downloaded then fail rate jumped to 51%, rate dropped to 0.2/s. Clearing 60-min cooldown and retrying re-triggered within 10 minutes.
**Root cause:** CDN rate-limit is a short rolling window (~2K requests/10-15 min), not a daily quota. Concurrency reduction alone doesn't help — volume throughput was still too high.
**Fix:** Added `--photo-delay FLOAT` argument to `download_cadastral_photos.py`. Sleeps N seconds between each individual photo download within a record (skips delay on first photo). Re-running VL with `--workers 4 --photo-delay 1.0` (~4 photos/sec across all workers = ~14K/hour, but with natural network jitter this should stay comfortably under the 2K/10-min threshold).
**Files:** `scripts/download_cadastral_photos.py`
**Recurrence:** First
