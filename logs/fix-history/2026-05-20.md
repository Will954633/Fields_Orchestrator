## [FP-001] Floor plan GPT analysis failed — Domain CDN returned 150×107px thumbnail — 22:38 AEST

**Symptom:** 12 Longueville Court had no room dimensions extracted. `floor_plan_analysis.rooms` was empty and `parsed_rooms` null despite `floor_plans_v2_extracted` containing a URL that appeared to be a 3277×2338 floor plan image.
**Root cause:** Domain CDN (`rimh2.domainstatic.com.au`) encodes a signed resize hash in the URL. The hash `7T4qYJN7-Rx38686gyk_vspyyhw=` resolves to a 150×107px thumbnail (21KB) even though the filename ends in `-w3277-h2338`. GPT-4o received a thumbnail too small to read any labels or dimensions.
**Fix:**
1. **Immediate data fix** — Updated `floor_plans_v2_extracted` for this property to use `bucket-api.domain.com.au/v1/bucket/image/<filename>` which returns the full 3028×2160px (522KB) image without a signed hash. Re-ran GPT-4o vision → extracted 12 rooms, 268sqm total floor area.
2. **Scraper fix (extract_floor_plans_from_v2_images.py)** — Added `_to_bucket_api_url()` helper. All `rimh2.domainstatic.com.au` URLs are now converted to `bucket-api` at write time, so `floor_plans_v2_extracted` always stores high-res URLs going forward.
3. **Reanalyze fallback (reanalyze_floor_plans_gpt54.py)** — Added `download_and_encode()` fallback: if downloaded image's shortest side < 500px, automatically retry with `bucket-api` equivalent URL.
**Files:** `scripts/extract_floor_plans_from_v2_images.py`, `scripts/reanalyze_floor_plans_gpt54.py`, DB update to `robina` collection.
**Recurrence:** First (but likely affects any property whose floor plan was classified after May 18 classifier update — only those properties where Domain CDN signed URL happened to encode a thumbnail)

## [SAT-001] Satellite annotation incorrectly labels neighbour's pool as subject property amenity — 21:40 AEST

**Symptom:** 12 Longueville Court showed "Pool" in the satellite DETECTED ON THE IMAGE panel. Pool is on the neighbour's lot, not the subject property. satellite_analysis.categories.amenity_premiums.pool_visible was correctly false, and the narrative correctly noted "no pool on subject lot", but the features array contained a "Pool" entry with a bbox clearly outside the subject lot boundary.
**Root cause:** The satellite annotation prompt in `satellite_annotation.py` did not constrain amenity features to the subject lot. GPT annotated any visible pool in the image frame, including neighbours'. The features array bbox [690, 350, 750, 410] vs subject bbox [560, 580, 720, 740] — completely different area.
**Fix:**
1. **Cadastral boundary system** — new `scripts/property_reports/lot_boundary.py` fetches the exact lot polygon from GCCC ArcGIS Cadastre_Current_view service using point-in-polygon query (no API key needed).
2. **Boundary draw + crop** — `satellite_annotation.py` now: (a) fetches the boundary polygon, (b) projects ring vertices to pixel space using Mercator math, (c) draws a yellow boundary outline on the image, (d) crops to the lot bbox + 1.5× margin so the subject property fills more of the frame.
3. **GPT prompt update** — updated to reference the visible yellow polygon: "ONLY annotate amenity features that fall INSIDE the yellow boundary polygon."
4. **Immediate data fix** — removed the erroneous Pool feature from 12 Longueville Court's features array in DB. Also added no-boundary fallback prompt for properties where GCCC lookup fails.
**Files:** `scripts/property_reports/lot_boundary.py` (new), `scripts/property_reports/satellite_annotation.py`, `scripts/property_reports/inline_satellite.py`
**Recurrence:** First (but likely affects many properties with pools on neighbouring lots)

## [CA-009b] VL photo retry — throttled photo-level rate limiting — 20:34 AEST

**Symptom:** Varsity Lakes photo downloads triggered Domain CDN rate-limit repeatedly even with `--workers 8` (reduced from 24). ~2K photos downloaded then fail rate jumped to 51%, rate dropped to 0.2/s. Clearing 60-min cooldown and retrying re-triggered within 10 minutes.
**Root cause:** CDN rate-limit is a short rolling window (~2K requests/10-15 min), not a daily quota. Concurrency reduction alone doesn't help — volume throughput was still too high.
**Fix:** Added `--photo-delay FLOAT` argument to `download_cadastral_photos.py`. Sleeps N seconds between each individual photo download within a record (skips delay on first photo). Re-running VL with `--workers 4 --photo-delay 1.0` (~4 photos/sec across all workers = ~14K/hour, but with natural network jitter this should stay comfortably under the 2K/10-min threshold).
**Files:** `scripts/download_cadastral_photos.py`
**Recurrence:** First
