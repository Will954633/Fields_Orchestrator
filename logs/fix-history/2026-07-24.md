## [INDEXING-NEW-PAGES-STARVED-BY-ORDERING] New editorial pages get near-zero organic traffic because the Indexing API job never submits them (oldest-first ordering under a 200/day cap) â€” 09:11 AEST
**Symptom (Will):** ~106 current-listing editorial pages published in the last 5 days (2026-07-19..24) were getting effectively no PostHog traffic. Asked why.
**Investigated the premise first (it was slightly off, in an informative way):** the pages are NOT untracked and NOT literally zero â€” 29/106 have real PostHog pageviews in the last 7 days (44 views total, 24 of them google/bing organic). So PostHog instrumentation on `/property/` is fine and some organic IS trickling in. The real finding is that per-page organic is near-zero because the pages are barely indexed, and off-market pages get ~4x the organic (89 google views/7d) despite being the same age â€” so "new listing pages aren't being surfaced by Google" is the actual problem, not tracking.
**Root cause â€” TWO compounding bugs in `google_indexing.py` (nightly `submit-new` cron, 01:00 AEST):**
1. *(found 2026-07-23, [GOOGLE-INDEXING-SUBMIT-NEW-NEVER-PERSISTS-STATE])* the last-run state file never persisted, so every night fell back to full-catalog submit. Tonight's 01:04 run was the FIRST to create `indexing-last-run.json` (confirmed: file now exists, content `2026-07-23T15:04Z` = 01:04 AEST). So the incremental path is still UNPROVEN â€” its first real exercise is the 2026-07-25 01:00 run.
2. *(found this session, NOT previously caught)* even in full-catalog mode, `get_property_urls()` returned URLs in `_id`-ascending (**oldest-first**) order, robina first, then the caller caps at DAILY_QUOTA=200. Robina alone has 699 URLs (97 for_sale + 602 sold) â€” so the 200/day quota was entirely consumed by ~37 static pages + the oldest ~163 robina URLs. Consequences: (a) genuinely NEW editorial pages sorted LAST and were never submitted; (b) varsity_lakes and burleigh_waters properties were **never reached at all**. Confirmed live in the log: `38-glen-eagles-drive-robina` hit `HTTP 429 quota exceeded` on tonight's run. This is the direct mechanism behind "we publish new pages and they get no traffic" â€” they were literally never submitted to Google's Indexing API; the only reason any get indexed is the sitemap (slower).
**Fix:** `get_property_urls()` now sorts NEWEST-first by `last_updated` (falling back to the ObjectId's embedded creation time), for-sale ahead of sold, across all three suburbs â€” so freshly-published listing pages are submitted first each night regardless of which suburb they're in.
**Verified (dry-run, no API calls â€” quota already exhausted today):** newly-published editorial pages appearing in the first-200 submit window went from ~0 to **75/106** (the remaining 31 have older `last_updated` and cycle in on following nights; and once bug-1's incremental path engages, nightly volume drops well under the 200 cap so all new pages submit promptly). `py_compile` clean.
**Files:** `scripts/google_indexing.py` (`Will954633/Fields_Orchestrator`, commit 318694b). Revert = revert that commit.
**Recurrence:** bug 2 is First. Note the honest limits for Will: this fixes SUBMISSION; it does not guarantee Google INDEXES a page (3 pages remain "Crawled - currently not indexed" for authority/quality reasons the API can't force), and a new low-authority domain has intrinsic crawl lag â€” so expect gradual improvement over days, not an overnight jump. Worth a Systems Health check that `indexing-last-run.json` is fresh AND that the incremental path actually engaged on 2026-07-25 (don't assume bug-1's fix works until that run is observed).
