## [INDEXING-NEW-PAGES-STARVED-BY-ORDERING] New editorial pages get near-zero organic traffic because the Indexing API job never submits them (oldest-first ordering under a 200/day cap) — 09:11 AEST
**Symptom (Will):** ~106 current-listing editorial pages published in the last 5 days (2026-07-19..24) were getting effectively no PostHog traffic. Asked why.
**Investigated the premise first (it was slightly off, in an informative way):** the pages are NOT untracked and NOT literally zero — 29/106 have real PostHog pageviews in the last 7 days (44 views total, 24 of them google/bing organic). So PostHog instrumentation on `/property/` is fine and some organic IS trickling in. The real finding is that per-page organic is near-zero because the pages are barely indexed, and off-market pages get ~4x the organic (89 google views/7d) despite being the same age — so "new listing pages aren't being surfaced by Google" is the actual problem, not tracking.
**Root cause — TWO compounding bugs in `google_indexing.py` (nightly `submit-new` cron, 01:00 AEST):**
1. *(found 2026-07-23, [GOOGLE-INDEXING-SUBMIT-NEW-NEVER-PERSISTS-STATE])* the last-run state file never persisted, so every night fell back to full-catalog submit. Tonight's 01:04 run was the FIRST to create `indexing-last-run.json` (confirmed: file now exists, content `2026-07-23T15:04Z` = 01:04 AEST). So the incremental path is still UNPROVEN — its first real exercise is the 2026-07-25 01:00 run.
2. *(found this session, NOT previously caught)* even in full-catalog mode, `get_property_urls()` returned URLs in `_id`-ascending (**oldest-first**) order, robina first, then the caller caps at DAILY_QUOTA=200. Robina alone has 699 URLs (97 for_sale + 602 sold) — so the 200/day quota was entirely consumed by ~37 static pages + the oldest ~163 robina URLs. Consequences: (a) genuinely NEW editorial pages sorted LAST and were never submitted; (b) varsity_lakes and burleigh_waters properties were **never reached at all**. Confirmed live in the log: `38-glen-eagles-drive-robina` hit `HTTP 429 quota exceeded` on tonight's run. This is the direct mechanism behind "we publish new pages and they get no traffic" — they were literally never submitted to Google's Indexing API; the only reason any get indexed is the sitemap (slower).
**Fix:** `get_property_urls()` now sorts NEWEST-first by `last_updated` (falling back to the ObjectId's embedded creation time), for-sale ahead of sold, across all three suburbs — so freshly-published listing pages are submitted first each night regardless of which suburb they're in.
**Verified (dry-run, no API calls — quota already exhausted today):** newly-published editorial pages appearing in the first-200 submit window went from ~0 to **75/106** (the remaining 31 have older `last_updated` and cycle in on following nights; and once bug-1's incremental path engages, nightly volume drops well under the 200 cap so all new pages submit promptly). `py_compile` clean.
**Files:** `scripts/google_indexing.py` (`Will954633/Fields_Orchestrator`, commit 318694b). Revert = revert that commit.
**Recurrence:** bug 2 is First. Note the honest limits for Will: this fixes SUBMISSION; it does not guarantee Google INDEXES a page (3 pages remain "Crawled - currently not indexed" for authority/quality reasons the API can't force), and a new low-authority domain has intrinsic crawl lag — so expect gradual improvement over days, not an overnight jump. ## [EDITORIAL-NUMBER-FORMAT] Editorial pipeline auto-publishes abbreviated $1.25m dollar formats — mandatory rule 5 violation on live pages — 09:35 AEST

**Root cause:** The editorial pipeline has NO deterministic number-format enforcement — the rule lives only in the LLM prompt, and the prompt was self-contradicting: three VOICE lines say "$1,250,000 not $1.25m" while ~20 few-shot examples across the generation AND fact-check-tolerance sections actively demonstrate/permit abbreviated formats (e.g. headline formula line "Comparable sales say low $1M… guide might say $1.4M"; verify agent told "$1.8M … is VERIFIED, rounding to nearest $100K is acceptable"). **Fix:**
1. Added a deterministic normalizer to `store_analysis` (`_expand_abbreviated_money` / `_normalize_money_formats`) that expands any `$X.Xm`/`$Xk` → full comma figures on the whole `ai_analysis` doc immediately before the DB write — a guaranteed safety net regardless of model output. Unit-tested (headlines, ranges, `$2M+`, unchanged full figures).
2. De-abbreviated the ~23 offending prompt example lines (generation + verify tolerance) to full figures, leaving the 3 negative-example lines (`not "$1.25m"`) intact.
3. Normalized the 3 already-stored docs in place using the same function (0 abbreviated tokens remain; statuses preserved: 2 published, 1 draft).
**Verified:** live SSR pages `/property/20-florabella-drive-robina` and `/property/288-christine-avenue-varsity-lakes` now serve full figures ($1,185,000; $1,200,000–$1,600,000) with zero abbreviated `$X.XM/$XK` tokens.

## [OFFMARKET-DECK-HYDRATION-418] Off-market deck threw React #418 (hydration text mismatch) from render-path Date.now() — 09:58 AEST
**Symptom:** PostHog `$exception` capture showed Minified React error #418 (`args[]=text` — server/client text hydration mismatch), unhandled, firing ONLY on `/off-market/*` deck pages — 3 distinct users on 3 addresses in 14 days (54 Heights Drive, 11 Botanique Terrace, 12 Orchid Tree Court, all Robina). Surfaced while tracing a "new lead" Telegram for 54 Heights Drive Robina: the tracked visitor (Google organic, iPhone/Mobile Safari, ~100s) hit the exception mid-session. When #418 fires React discards the SSR subtree and re-renders client-side → content flash/reflow on the highest-value organic owner-lookup landing.
**Root cause:** `computeGainVerdict` in `ladderShared.ts` called `Date.now()` DURING render (`const years = (Date.now() - purchaseMs) / YEAR_MS_V`). The deck is SSR'd then hydrated, so the function runs twice — once on Netlify at request time, once in the browser at hydration time — sampling different instants. `years` feeds the CAGR figures (`pct1(...)`, rounded to 1 dp) and the verdict tier branch (`if (propCAGRLow > bankCAGR) …`). When a property's numbers sit on a rounding edge or tier boundary, the µs delta flips the rendered text between server and client → #418. Rare + data-dependent because only boundary-value properties flip (why 3 users, not every view). NOTE: `MONTH_YEAR` (`toLocaleDateString`) was the first suspect (see the toLocaleDateString hazard in [HOUSE-MINI-SITE-003]) but was correctly ruled out — the sale dates are date-only / naive (no-Z) timestamps, for which the UTC↔AEST shift never crosses a month boundary at month/year granularity.
**Fix:** Capture a single wall-clock instant server-side in the route loader (`off-market.$slug.tsx` → `property.asOfMs = Date.now()`), thread it via the serialized loader data (reused verbatim on hydration), and have `computeGainVerdict` use `data.asOfMs ?? Date.now()` instead of calling `Date.now()` itself. SSR and hydration now compute against one identical instant, so the whole class of divergence is gone, not just the one field. Added `asOfMs?: number` to the `PropertyData` type. Verified: `react-router build` clean; live deck (commit 191f802, Netlify deploy `ready`) screenshots with console 0 msgs / 0 network errors, gain-verdict block renders correctly ("excellent result… 14.8%–19.3% a year… term deposit around 3.3%… inflation around 4.5%").
**Files:** Feilds_Website/01_Website/src/pages/OffMarketPage/types.ts, Feilds_Website/01_Website/src/pages/OffMarketPage/ladderShared.ts, Feilds_Website/01_Website/src/routes/off-market.$slug.tsx
**Recurrence:** 3rd React #418 hydration-mismatch (after [HOUSE-MINI-SITE-003] toLocaleDateString comma, [HOUSE-MINI-SITE-006] useState(()=>location.hash)). Pattern now firmly established: ANY non-deterministic value read during render — `Date.now()`/`new Date()`, `location.hash`/`window.*`, locale/timezone-dependent formatting — is an SSR hydration hazard. Rule: compute time/env-dependent values once in the loader (server) and thread them as data; seed client-only state deterministically and sync in useEffect. Still-open sibling errors seen in the same PostHog sweep (NOT fixed here): `window.webkit.messageHandlers` undefined (29 users, likely in-app-browser injection) and React #185 max-update-depth loop (2 users).
