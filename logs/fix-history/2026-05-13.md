## [AKAMAI-BLOCK-001] Domain.com.au scraping silently failing — Akamai bot block + missing error propagation — 09:10 AEST

## [BACKUP-SCRAPER-002] Backup scraper Phase 2 — cadastral sync + abbreviation-aware norm — 23:55 AEST
**Symptom:** After Phase 1 backfill, coverage check showed Robina at 66% address-in-baseline (48/73 production for_sale addresses). Most "missing" were actually present in backup but stored as `27 BEAUTY POINT DRIVE` (cadastral) vs `27 Beauty Point Drive, Robina, QLD 4226` (JSON discovery) and `5 Peppertree Circuit` vs `5 PEPPERTREE CIRCUT` (typos). Also: backup VM's Nov 2025 cadastral snapshot was 80 records behind production for Robina (46 missing) + 99 behind for Varsity Lakes + 50 behind for Burleigh Waters.

**Root cause:** Two issues compounding the apparent gap:
1. **Cadastral snapshot drift.** Backup VM's local Mongo `Gold_Coast.{suburb}` cadastral records were imported Nov 6, 2025. Production has since added ~200 new addresses (mostly strata unit numbers + small subdivisions). Backup cadastral universe was a subset of production.
2. **Abbreviation mismatch.** Cadastral always uses full street types (`AVENUE`, `STREET`, `DRIVE`, `CIRCUIT`). JSON-scraped addresses use whatever the agency writes — usually short form (`Ave`, `St`, `Dr`, `Cct`). The Phase 1 norm function only handled commas and case; abbreviations broke matching for ~10-20% of records.

**Fix:**
1. **`scripts/sync_cadastral_from_production.py`** (new — 220 lines, two-VM workflow). Run with `--export` on this VM to dump cadastral-only fields (no Domain-derived data) from production Cosmos to a JSON bundle in `/tmp`. Bundle is then SSH-copied to backup VM and applied with `--apply`, which loads existing local addresses into memory once and bulk-upserts. Production cadastral fields strictly limited to non-Domain-derived columns (`ADDRESS_PID`, lat/lng, `lot_size_sqm`, `STREET_*`, `LOCALITY`, etc.) — backup remains genuinely independent of Domain scraping for all listing data. Backup VM Robina now has 12,196 docs (11,823 cadastral + 373 listings-only stubs), VL 7,701 (7,670 + 31), BW 6,900 (6,895 + 5).

2. **Abbreviation-aware `_normalise_address()` across all 4 scripts.** 21-entry `STREET_ABBREV` map (`AVE→AVENUE`, `RD→ROAD`, `DR→DRIVE`, `CT→COURT`, `PL→PLACE`, `CRES→CRESCENT`, `CCT→CIRCUIT`, `WY→WAY`, `BLVD→BOULEVARD`, `TCE→TERRACE`, `HWY→HIGHWAY`, `PDE→PARADE`, `CL→CLOSE`, `LN→LANE`, `GR→GROVE`, plus less-common variants). Expansion fires only when the abbreviation appears in the non-terminal middle (typically position `len-4`) — so `ST IVES DRIVE` stays as `ST IVES DRIVE` (the `ST` is a proper noun, position 1, not expanded), while `5 BEACH PL ROBINA QLD 4226` becomes `5 BEACH PLACE ROBINA QLD 4226`. Applied to `url_tracker.py`, `ingest_json_to_mongo.py`, `remediate_cadastral_norm.py`, `sync_cadastral_from_production.py`.

## [BACKUP-SCRAPER-003] Phase 3 — field extraction quality (HybridExtractor) — 01:10 AEST
**Symptom:** Per-property field completeness audit (12 currently-matched for_sale Robina properties, backup vs production) showed six fields the backup scraper was NOT extracting at all even though production captures them at near-100%:
- `agent_name` / `agent_names` (0% vs 100%)
- `price` (textual price guidance like "Contact Agent" / "Offers Over $1.2M" / "Auction Sat 18 May") (0% vs 100%)
- `inspection_times` (0% vs 92%)
- `first_listed_date` (0% vs 100%)
- `total_floor_area` (internal m², distinct from land) (0% vs 83%)
- `floor_plans` (separate gallery, 0% vs 92%)

The pre-existing `description` field was being populated with the first 500 chars of body text — which on agency sites is mostly nav/header noise, not the agent's prose writeup.

**Root cause:** The `HybridExtractor` was built before the field set was finalised against production. It captured the obvious things (beds/baths/cars/price-numeric/property_type/features/listing_status) but never had extractors for the agent block, inspection schedule, listed date, internal floor area, or price-text.

**Fix:** Six new extractors added to `RuleBasedExtractor` (all rule-based, no LLM cost) + a new `extract_description()` that carves out the prose between `Property for Sale` and footer markers like `Property Features` / `Outgoings` / `Show More` / `Map Location`. Expanded `feature_keywords` from 12 to 30+ entries (Solar, Water tank, Fireplace, Walk-in pantry, Stone benchtops, Hardwood floors, Waterfront, North-facing, etc.).

1. **`extract_price_text(text, title)`** — multi-pattern matcher for "Under Contract", "Auction Sat 18 May", "Offers Over $X", "Contact Agent", "POA", "EOI Closing", plain "$1,250,000 Negotiable". Pattern ordering matters — strong markers (Under Contract, Auction with-explicit-date) checked before the catch-all numeric.
2. **`extract_agents(text)`** — heuristic. Look for title-cased 2-4-word names where (a) every word is alpha + starts uppercase + has a lowercase char (rejects "07 5578 8800" and "UNDER CONTRACT"); (b) line doesn't contain any agency hint (excludes "Ray White Robina" as a candidate); (c) within 1-3 lines after the candidate there IS an agency hint (confirms it's the agent for THAT agency). 50-entry stopword list (months, common headers, "You", "Might", "Also", "Like", "Past", "Recent", "Latest" etc.) prevents picking up section headers.
3. **`extract_inspection_times(text)`** — collapses newlines first (agency sites often stack each token: `Saturday\n4\nApril\n10:30am - 11:00am`), then matches `Weekday + day + month [+ year] + time-range`.
4. **`extract_first_listed_date(text)`** — matches "Added DD Month, YYYY", "Listed on DD/MM/YYYY", "Posted DD Month YYYY", "Date Added" variants. Returns ISO format.
5. **`extract_total_floor_area(text)`** — multi-pattern matcher for "internal living", "internal area", "floor area", "under roof", "approx. 110m² layout", and "Building: 362m²". Sanity bounds 30-2000 m².
6. **`extract_floor_plans(images)`** — pulls floor-plan URLs out of the raw images list by alt-text or URL pattern (`floorplan`, `floor-plan`, `/fp/`, `_fp_`).

Wired into `extract_property_data()` → `create_mongodb_document()`, plus extended `SNAPSHOT_FIELDS` in `url_tracker.py` so the new fields land on the cadastral property doc under `backup_scraper.*`.

**Cadastral fallback** added to `URLTracker._upsert_property_snapshot()` — if a listing doesn't expose `land_size_sqm` but the cadastral doc has one, copies it onto the snapshot as `backup_scraper.land_size_sqm_from_cadastral`. Also stamps `backup_scraper.latitude` / `.longitude` from cadastral so downstream consumers don't have to join.

**`scripts/reprocess_discoveries.py`** (new — 130 lines). Re-runs the updated extractor against every existing `new_url_discoveries` record with raw_data, refreshing the extracted_data field AND updating the cadastral snapshot — gives instant benefit from extractor improvements without waiting for a full re-scrape. Dedupes by `complete_address_norm` so each address is processed once (using the latest discovery). Idempotent. Stamps `extractor_version: phase3_2026_05_13` for traceability.

## [HOUSE-MINI-SITE-003] React #418 hydration warning on /your-home — 10:08 AEST
**Symptom:** Site inspector caught React minified error #418 (hydration mismatch) on every load of the seller mini-site. **Recurrence:** First. Note for future: `toLocaleDateString` without a fixed format string is a hydration hazard inside any SSR component. Numeric `toLocaleString("en-AU")` is usually safe (thousand separators consistent), but date formatting is not.

## [HOUSE-MINI-SITE-006] Hydration warning persisted after date-formatter fix — hash-driven tab state — 10:48 AEST
**Symptom:** After deploying the deterministic date formatter (HOUSE-MINI-SITE-003), React #418 hydration mismatch was still firing on `/your-home/...#market`. The /home tab (no hash) was clean — hash-bearing URLs still warned.
**Root cause:** `useState<TabId>(() => parseTab(location.hash))` reads `location.hash` during the initial render. On Netlify SSR there's no hash (server emits "home" tab); on the client the hash is `#market` (browser emits "market" tab). Different initial tree → React refuses hydration, falls back to client re-render.
**Fix:** Always seed `activeTab = "home"` on the initial render. The existing `useEffect` that listens to `location.hash` already syncs the tab on client mount and on hash changes, so the user sees the correct tab a tick after hydration with no flicker (the layout above the tab content is identical across tabs).
**Files:** `01_Website/src/pages/YourHomePage/YourHomePage.tsx`
**Recurrence:** First (related to 003 — same class of bug, second instance). Pattern note: any `useState(() => readWindowOrLocation())` initialiser is a hydration trap. Default to a deterministic seed value, sync via useEffect.
