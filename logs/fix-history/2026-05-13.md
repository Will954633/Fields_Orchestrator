## [AKAMAI-BLOCK-001] Domain.com.au scraping silently failing — Akamai bot block + missing error propagation — 09:10 AEST
**Symptom:** Website data appeared stale for 2+ days. Investigation showed:
- Step 101 reported `success: true, exit_code: 0` for the past 3 nightly runs (May 11, 12 — May 13 not yet run)
- Actual stdout: `[Robina] Page 1: Failed to fetch, stopping pagination` for ALL target suburbs, 0 URLs discovered
- Direct probe from VM (IP `34.40.230.132`) returned HTTP 403 from Domain on every path (homepage, `/sale/`, `/sold-listings/`, `/property-profile/`, even `/robots.txt`)
- Response header: `server: AkamaiGHost`. Body was the Akamai "Behavioral Challenge" interstitial with `Powered and protected by Akamai`
- Pattern across last 60 runs: scraping was already intermittent — multi-day blocks would engage after each Sunday all-suburbs run (step 102, ~2,700 URLs), then thaw by Friday. New in May 2026; April Sundays didn't trigger the block.
- The same block hit the backup property-scraper VM (`35.201.6.222`) — Akamai had flagged both GCP IPs.

**Root cause:** Two compounding issues:
1. **Akamai block trigger.** The Sunday all-suburbs run (step 102, 81 suburbs, ~2,700 URLs in one session) was tipping Akamai's bot management thresholds, causing Mon-Thu blocks on this VM's IP. Direct `curl_cffi` with chrome120 TLS impersonation couldn't bypass — Akamai's interstitial requires JS execution to solve.
2. **Silent-failure bug in step 101.** `run_curlffi_suburb_scrape.py:main()` returned 0 even when every suburb discovered 0 URLs. The orchestrator + watchdog therefore saw "✅ success" and didn't page anyone. Stale data sat for 2 weeks before being noticed.

**Fix:** Five-part repair, all live + on GitHub:

1. **Bright Data Web Unlocker integration (primary fix).** Added Will's existing `web_unlocker2` zone as the fetch layer for every script that hits Domain HTML. New helper `shared/domain_fetch.py` exposes `fetch_html(url)` and `fetch_with_status(url)` — both POST to Bright Data's API which solves the Akamai JS challenge and returns real content. Falls back to direct `curl_cffi` when `BRIGHTDATA_API_KEY` is unset. Cost: ~$0.75/day for target market step 101.

2. **Disabled Sunday all-suburbs run.** Set `schedule.run_other_suburbs_weekly: false` in `config/settings.yaml`. Steps 102 + 104 will not run; per memory, non-core suburbs only need sample-level coverage for comparison work, not 100% completeness. Will revisit if/when bandwidth allows splitting them across the week.

3. **Step 101 silent-success fix.** Added a final check in `main()`: if `total_discovered == 0` across all suburbs, exit code 2 with a loud error message. The orchestrator/watchdog now surfaces this as a failure instead of greenlighting through.

4. **Patched 4 downstream scripts to use the helper:**
   - `sold_backfill/search_based_sold_monitor.py` (step 103) — `fetch_page()` now calls `fetch_html()`
   - `scripts/coverage_check.py` (step 109) — both `fetch_domain_count()` and `fetch_domain_sold_count()` use helper
   - `scripts/scrape_recent_sold.py` (step 111) — same pattern as 103
   - `scripts/detect_withdrawn.py` (step 113) — reworked `check_listing_status()` to use content-aware detection (title-based "Sold ..." check, body markers for "off the market") instead of relying on HTTP 301 redirects (Domain no longer redirects withdrawn listings — they serve 200 with state in title/body)

5. **Confirmed step 110 (image download) is unaffected** — `rimh2.domainstatic.com.au` and `bucket-api.domain.com.au` return 200 from this VM directly. Domain's image CDNs are not behind Akamai bot management.

## [BACKUP-SCRAPER-001] Backup scraper zombie — Cosmos dead, no throttling, no local persistence — 23:30 AEST
**Symptom:** Backup property scraper on `35.201.6.222` had been running 66 days with no useful output:
- 13,220+ cycles, 79,639 JSON files on disk (2.9GB), but only 2 unique addresses being re-scraped per cycle for the past month
- Cosmos DB `Gold_Coast_Backup.property_url_tracking` returning 403 Forbidden on every call (Azure subscription disabled)
- RE/MAX GC + Coastal newly returning 403 (IP-fingerprinted after 2 months continuous scraping)
- McGrath still hard-blocked (Vercel JS challenge — known)
- `scraper.log` at 132MB, no rotation
- Cycle interval ~7 min = 200 cycles/day, contributing to agency IP blocks
- Production has 73 for_sale Robina properties; backup had no way to even compare

**Root cause:** Three compounding failures:
1. **Cosmos subscription dead.** URLTracker writes failed → `detect_new_urls` returned all URLs as "new" → tried to scrape → couldn't write → infinite retry loop on the same handful of records that were already in memory.
2. **No local persistence layer.** Even with rich JSON output (79K files, including 5,199 unique Robina addresses with property data), nothing was being merged into MongoDB. Backup was a JSON-write-only system, useless as a query target.
3. **No throttling, no UA rotation.** Continuous 7-min cycles with a single hardcoded user-agent string had trained agency WAFs to recognise and block the scraper.

**Fix:** Phase 0 hardening + Phase 1 local Mongo replacement (Phases 2-7 deferred to follow-up sessions per agreed workflow):

**Phase 0 — Hardening (4 changes):**
1. **Throttle.** `continuous_monitor.py` now sleeps `CYCLE_INTERVAL_SECONDS = 1800` (30 min) between cycles, env-overridable. Cuts daily request volume from 200 cycles/day to 48.
2. **User-agent rotation.** `robust_extractor.py` rotates between 6 realistic recent Chrome/Firefox UAs (macOS + Windows) per request.
3. **Per-request jitter.** 0.5-3.0s random sleep before each `page.goto()` so concurrent requests don't fire in perfect lockstep.
4. **Log rotation.** Installed `/etc/logrotate.d/property-scraper` (daily, 7 rotations, compress, copytruncate) on backup VM. Truncated current 132MB log.

**Phase 1 — Local Mongo replacement (3 components):**
1. **Rewrote `url_tracker.py`** — drop-in replacement for the Cosmos-backed version. Same public API (`get_known_urls`, `detect_new_urls`, `update_tracking`, `record_discovery`, `mark_json_saved`) so `continuous_monitor.py` is unchanged. Backed by local `Gold_Coast.property_url_tracking` + `Gold_Coast.new_url_discoveries` + denormalized snapshot fields on `Gold_Coast.{suburb}` cadastral docs (`backup_scraper.*` namespace + top-level `listing_status` + `scrape_source: "backup_scraper"`).
2. **JSON → Mongo backfill (`scripts/ingest_json_to_mongo.py`).** Walks `discovered_urls/{suburb}/*.json`, dedupes by address, picks the latest-strongest status per address, upserts into the three collections. Idempotent.
3. **Cadastral normalization remediation (`scripts/remediate_cadastral_norm.py`).** JSON discovery addresses have commas (`33 Trinity Place, Robina, QLD 4226`) but cadastral baseline has none (`33 ST IVES DRIVE ROBINA QLD 4226`). Added `complete_address_norm` field to all cadastral docs (uppercase, no commas, single-spaced). 1,107 listings-only stubs after first backfill were re-matched, 772 merged into cadastral.

**Verification:**

## [BACKUP-SCRAPER-002] Backup scraper Phase 2 — cadastral sync + abbreviation-aware norm — 23:55 AEST
**Symptom:** After Phase 1 backfill, coverage check showed Robina at 66% address-in-baseline (48/73 production for_sale addresses). Most "missing" were actually present in backup but stored as `27 BEAUTY POINT DRIVE` (cadastral) vs `27 Beauty Point Drive, Robina, QLD 4226` (JSON discovery) and `5 Peppertree Circuit` vs `5 PEPPERTREE CIRCUT` (typos). Also: backup VM's Nov 2025 cadastral snapshot was 80 records behind production for Robina (46 missing) + 99 behind for Varsity Lakes + 50 behind for Burleigh Waters.

**Root cause:** Two issues compounding the apparent gap:
1. **Cadastral snapshot drift.** Backup VM's local Mongo `Gold_Coast.{suburb}` cadastral records were imported Nov 6, 2025. Production has since added ~200 new addresses (mostly strata unit numbers + small subdivisions). Backup cadastral universe was a subset of production.
2. **Abbreviation mismatch.** Cadastral always uses full street types (`AVENUE`, `STREET`, `DRIVE`, `CIRCUIT`). JSON-scraped addresses use whatever the agency writes — usually short form (`Ave`, `St`, `Dr`, `Cct`). The Phase 1 norm function only handled commas and case; abbreviations broke matching for ~10-20% of records.

**Fix:**
1. **`scripts/sync_cadastral_from_production.py`** (new — 220 lines, two-VM workflow). Run with `--export` on this VM to dump cadastral-only fields (no Domain-derived data) from production Cosmos to a JSON bundle in `/tmp`. Bundle is then SSH-copied to backup VM and applied with `--apply`, which loads existing local addresses into memory once and bulk-upserts. Production cadastral fields strictly limited to non-Domain-derived columns (`ADDRESS_PID`, lat/lng, `lot_size_sqm`, `STREET_*`, `LOCALITY`, etc.) — backup remains genuinely independent of Domain scraping for all listing data. Backup VM Robina now has 12,196 docs (11,823 cadastral + 373 listings-only stubs), VL 7,701 (7,670 + 31), BW 6,900 (6,895 + 5).

2. **Abbreviation-aware `_normalise_address()` across all 4 scripts.** 21-entry `STREET_ABBREV` map (`AVE→AVENUE`, `RD→ROAD`, `DR→DRIVE`, `CT→COURT`, `PL→PLACE`, `CRES→CRESCENT`, `CCT→CIRCUIT`, `WY→WAY`, `BLVD→BOULEVARD`, `TCE→TERRACE`, `HWY→HIGHWAY`, `PDE→PARADE`, `CL→CLOSE`, `LN→LANE`, `GR→GROVE`, plus less-common variants). Expansion fires only when the abbreviation appears in the non-terminal middle (typically position `len-4`) — so `ST IVES DRIVE` stays as `ST IVES DRIVE` (the `ST` is a proper noun, position 1, not expanded), while `5 BEACH PL ROBINA QLD 4226` becomes `5 BEACH PLACE ROBINA QLD 4226`. Applied to `url_tracker.py`, `ingest_json_to_mongo.py`, `remediate_cadastral_norm.py`, `sync_cadastral_from_production.py`.

## [HOUSE-MINI-SITE-002] titleFromSlug bug — "Terrace Court" double-match — 09:55 AEST
**Symptom:** Browser tab title for `/your-home/13-terrace-court-merrimac` rendered as "13 Terrace, Court Merrimac · Property Report" instead of "13 Terrace Court, Merrimac · Property Report".
**Root cause:** `titleFromSlug` scanned the slug parts left-to-right looking for a `STREET_TYPES` match. The street name "Terrace Court" contains both "terrace" AND "court" as street-type words; the left-scan picked "terrace" (at index 1) as the split point, putting the comma after "Terrace" and leaving "Court Merrimac" as the suburb.
**Fix:** Scan right-to-left instead. The actual descriptor (the trailing street-type word) wins. Also handles multi-word suburbs like "Burleigh Waters" correctly.
**Files:** `01_Website/src/routes/your-home.$slug.tsx`
**Recurrence:** First. Edge case I should have caught while writing the function — "Terrace Court Merrimac" is the build template slug.
