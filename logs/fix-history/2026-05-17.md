# Fix History — 2026-05-17

## [OOM-CHANGE-DETECTOR] VM OOM-killed mongod mid-nightly-run, SSH unreachable — 09:20 AEST
**Symptom:** Overnight 2026-05-16 the kernel OOM-killer terminated mongod at 21:34:40 AEST during the orchestrator's 20:30 run. GCP reported the VM "Running" but every port was unreachable (sshd also starved). Recovered via `gcloud compute instances reset` (hard reset performed from local-machine Claude). Pattern goes back further: runs on 2026-05-13/14/15 all silently failed with the same DB pressure, just without taking mongod fully down.
**Root cause:** `src/property_change_detector.py:104` did `col.find({}, {})` — no filter, no projection — across every target-suburb collection. Two uncommitted edits (`FOR_SALE_DATABASE = "Gold_Coast_Currently_For_Sale"` → `"Gold_Coast"`) redirected this scan from the deprecated active-only collections (~270 docs each) onto the unified Gold_Coast collections, which now hold ~40K cadastral records per suburb. Measured: 38,100 docs × ~205 KB = ~7.6 GB raw payload pulled into a Python list each run. At OOM time the offending Python process held 2.29 GB RSS; mongod (2.40 GB) + this process + a 1.8 GB node session = ~6.5 GB on an 8 GB VM. OOM-killer picked mongod. Same redirect was made in `src/unknown_status_detector.py` — that file already uses projections so it didn't OOM, but it lost semantic correctness: every cadastral record was being treated as an "unknown status" candidate.

Local-machine Claude's recovery diagnosis blamed fields-trigger-poller hammering disabled Azure Cosmos every 30s. That was incorrect — `config/settings.yaml` was switched to `mongodb://localhost:27017/` on 2026-04-26 and the poller has been talking to local mongod since. The 403 entries it saw in `logs/trigger-poller.log` are stale from April. Local Claude disabled the poller on this false premise; re-enabled here.

**Fix:**
1. `src/property_change_detector.py:_get_suburb_docs` — added `{"listing_status": "for_sale"}` filter and projection of `TRACKED_FIELDS + address + url`. Robina payload measured: 2.4 GB → 30 KB. Full-suburb total: ~7.6 GB → ~200 KB.
2. `src/unknown_status_detector.py` — added `{"listing_status": "for_sale"}` filter to both the pre-Phase 2 snapshot scan and the post-Phase 2 current-listings scan. Restores semantic correctness on the unified collections.
3. Drop-in `/etc/systemd/system/fields-orchestrator.service.d/memory-limits.conf` — `MemoryHigh=3500M`, `MemoryMax=4500M`. Future memory bugs now kill the orchestrator instead of mongod.
4. `/etc/mongod.conf` — set `storage.wiredTiger.engineConfig.cacheSizeGB: 1.5` (was implicit default ~3.5 GB on 8 GB VM). Backup at `/etc/mongod.conf.bak.20260517`. Verified live: `serverStatus().wiredTiger.cache."maximum bytes configured" = 1.50 GB`.
5. `sudo systemctl enable --now fields-trigger-poller` — undo local Claude's mistaken disable.
