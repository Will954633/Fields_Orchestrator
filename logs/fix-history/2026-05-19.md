## [HOUSE-WEBSITE-BUILD-PROGRESS-3] Live build feed was static — stale poller modules + completion bug — 13:25 AEST

**Symptom:** Will submitted "3 Lumley Street, Mudgeeraba" and landed on the new building page. Page rendered correctly but **stayed at 0% forever, with no steps populating**. Click-through to /your-home eventually worked (resolver finished, state → under_review) but the live build feed never animated.

**Root cause (two compounding bugs):**

1. **`fields-property-report-poller` had stale Python modules.** The systemd service is a long-running Python process that imports `scripts.property_reports.slot_resolver` + `build_property_report` once at startup. My instrumentation edits landed AFTER the poller started, so the running process kept executing the pre-instrumentation code path — `build_state` and `build_events` were never set, and the progress endpoint returned `events: []` indefinitely. Confirmed via DB inspection: doc had `state: "under_review"` (resolver completed) but `build_state: None, build_events: 0`.

2. **Completion-pct would plateau below 100% for thin-data properties.** Even after fix #1, properties without notable features legitimately skip the gallery/positioning/personas/buyers/scarcity_story steps (those resolver blocks are conditional on `scarcity_struct.get("notable_features")` and `clean_candidates` being non-empty). With only 8 of 13 canonical steps firing, the progress bar would max out at ~61% even on a `ready` build — looks broken even when it isn't.

**Fix:**

1. `sudo systemctl restart fields-property-report-poller` — picked up the new build_events emitter code.
   - Verified by force-resolving `3-lumley-street-mudgeeraba`: build_state transitioned stub → building → complete in 32s, with 14 events spanning 8 unique steps (the 5 missing steps were correctly skipped per the conditional branches).

2. `netlify/functions/property-report-progress.mjs` — `completion_pct = ready ? 100 : (done / total) * 100`. When the resolver has flipped build_state to "complete", the percentage shows 100 regardless of skipped steps.

3. `src/pages/AnalyseYourHomePage/BuildingPage.tsx` — `phaseFor()` now returns `"skipped"` (instead of `"pending"`) for any step that has no event AND the build is complete. The frontend strikes the label through with a faded icon, signalling "this stage didn't apply to your property" rather than "we're still working on it".

4. `src/pages/AnalyseYourHomePage/BuildingPage.module.css` — added `.phase_skipped` styling (line-through label, near-transparent icon).
