## [SECRET-LEAK-ROTATE] Move mongod creds out of unit files + rotate password — 09:10 AEST

**Symptom:** Carried over from 2026-05-17 OOM postmortem (P2.7). `Environment="COSMOS_CONNECTION_STRING=mongodb://fieldsadmin:<password>@..."` in `/etc/systemd/system/fields-orchestrator.service` and `fields-claude-agent.service` was parsed by systemd at every boot and the value landed in the GCP serial console / boot log, then into GCP's log retention. Same `fieldsadmin` credential was hardcoded in `scripts/mongodb-backup.sh`. Mongod also exposed on `0.0.0.0:27017` (Netlify Functions in AWS connect remotely), so the leaked credential was actually a live attack vector — not stale.

**Root cause:** Pre-existing pattern of embedding `Environment=` in unit files instead of `EnvironmentFile=`. Most fields-* services already used `EnvironmentFile=/home/fields/Fields_Orchestrator/.env`; orchestrator + claude-agent were the holdouts.

**Fix (two-phase):**

*Phase A — stop the leak:*
1. Created drop-in compatible refactor of both unit files: removed `Environment="KEY=VALUE"` lines, added `EnvironmentFile=/home/fields/Fields_Orchestrator/.env`. Verified all 5 keys (COSMOS_CONNECTION_STRING, AZURE_STORAGE_CONNECTION_STRING, BRIGHTDATA_API_KEY, BRIGHTDATA_ZONE, ANTHROPIC_API_KEY) already present in .env with matching values. Backups at `*.bak.20260519`.
2. `daemon-reload + restart fields-orchestrator` — verified all 5 vars present in running process env via `/proc/$PID/environ`. claude-agent disabled/inactive, will pick up .env on next start.

*Phase B — invalidate the leaked credential:*
1. Inventory: 8 password references across 5 files — `.env` (×2), `config/settings.yaml`, 3 website `.env` files, `scripts/mongodb-backup.sh`. Plus 1 Netlify env var (COSMOS_CONNECTION_STRING on site `lambent-tapioca-86ef75` / fieldsestate.com.au, scope `all`, pointing at VM public IP `34.40.230.132`).
2. Generated new 40-char URL-safe password via `secrets.token_urlsafe`. Created new mongod user `fields_app` with `root@admin` role (matching `fieldsadmin`). Kept `fieldsadmin` alive during cutover so external clients didn't break.
3. `sed -i 's|fieldsadmin:OLDPASS|fields_app:NEWPASS|g'` across all 5 local files. Backups at `*.bak.20260519`.
4. Patched `scripts/mongodb-backup.sh`: removed the hardcoded `MONGO_URI=` literal, now sources `$COSMOS_CONNECTION_STRING` from `.env`. Also added `trap cleanup_dump_dir EXIT` + moved `rm -rf $DUMP_DIR` before the upload step — addresses the orphan-dump bug from 2026-05-17 P0/P1 work that left ~38 GB of cruft when GCS uploads failed.
5. Restarted all 13 active `fields-*.service` units in sequence — all came back active/running, no auth failures in mongod.log.
6. Updated Netlify env var via API (PUT `/api/v1/accounts/{slug}/env/{key}`); triggered build-hook redeploy.
7. Confirmed via mongod.log: external auths from AWS IPs (Netlify) switched from `fieldsadmin` → `fields_app` within ~3 min after redeploy. Zero `fieldsadmin` auths in trailing 60s.
8. `db.dropUser("fieldsadmin")`. Verified the old credential string now returns `Authentication failed.` against localhost.

## [BACKUP-CRUFT-CLEANUP] Remove 38 GB of orphaned mongodump dirs + script patch — 09:15 AEST

**Symptom:** Carried over from 2026-05-17 OOM postmortem (P2.6). `/home/fields/backups/mongodb` was 46 GB and disk at 88%. Contents: 8 uncompressed `dump_2026-04-{13,17..23}_0200` directories at 4 GB each (~32 GB) and one `dump_2026-05-17_0200` at 6.6 GB. The April dumps predate the GCS bucket's earliest retained backup (April 26), so they were the *only* copies of pre-Cosmos-migration state.

**Root cause:** `scripts/mongodb-backup.sh` runs `mongodump → tar → upload → rm -rf $DUMP_DIR` under `set -e`. When the GCS upload failed (DNS error on 2026-05-17, auth/transient errors during the April migration window), `set -e` aborted before the `rm`, leaving the dump dir behind. Today's stranded one came from the upload step failing as the VM was crashing on 2026-05-16/17.

**Fix:**
1. Pushed today's existing compressed tarball (`fields_mongodb_2026-05-17_0200.tar.gz`, 1.1 GB) to GCS, then removed `dump_2026-05-17_0200` (freed 6.6 GB).
2. For each of the 8 April dump dirs: `tar -czf archive_<date>.tar.gz`, uploaded to `gs://fields-mongodb-backups/archive/`, verified GCS size matches local, then deleted both the local archive and the dump dir. All 8 now exist in GCS at `gs://fields-mongodb-backups/archive/archive_2026-04-{13,17..23}_0200.tar.gz` (~690 MB each, ~5.5 GB total in GCS).
3. Patched `scripts/mongodb-backup.sh`: (a) added `trap cleanup_dump_dir EXIT` that always removes `$DUMP_DIR` if it still exists; (b) moved the `rm -rf "$DUMP_DIR"` to immediately after compression succeeds (before upload) so a failed upload no longer leaves cruft behind. Verified `bash -n` and `--dry-run` pass.

**Disk impact:** 86% used → 48% used. Free space 13 GB → 51 GB.

**Files:**
- `scripts/mongodb-backup.sh` (pushed to Will954633/Fields_Orchestrator — single commit covers this + the .env-sourcing change from SECRET-LEAK-ROTATE)
- GCS objects under `gs://fields-mongodb-backups/archive/` (8 new)

**Recurrence:** First time the cleanup happened; the script patch prevents recurrence.
