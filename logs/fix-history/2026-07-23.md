## [QUARTER-COMPLETENESS-FALSE-POSITIVE] Systems Health flagged a working fix as broken — direction-blind comparison — 03:35 AEST
**Context:** A loop cycle's Task 0.5 sweep found 3 new ERROR rows ("Indexed prices — quarter completeness", Burleigh Waters/Robina/Varsity Lakes) that hadn't appeared in earlier cycles' checks that same session.
**Symptom:** `raw sold data has 52 sales in Q1 2026 but the precomputed series' latest complete quarter is Q2 2026` — read at a glance like a regression.
**Root cause:** the check (added earlier today per `[SYSTEMS-HEALTH-CONSOLIDATION]`/`[PROPERTY-TIMELINE-REFRESH-DEAD]`, see `2026-07-22.md`) computes an `expected_label` two quarters back from now (a grace window, since `property_timeline` is weekly-refreshed and `indexed_prices` monthly-precomputed) and flags ERROR whenever `latest_complete_period != expected_label`. Plain inequality doesn't distinguish "behind schedule" from "ahead of schedule" — and this morning's `[PROPERTY-TIMELINE-REFRESH-DEAD]` fix had, by this evening, genuinely caught the precomputed series up to Q2 2026 — ONE quarter ahead of the check's own 2-quarter grace expectation. The check flagged the exact outcome the morning's fix was meant to produce as an error.
**Fix:** `main_site_health_check.py`'s quarter-completeness check now parses both periods into `(year, quarter)` tuples and only flags ERROR when the precomputed series is genuinely BEHIND the expected quarter (`<`), never merely not-equal. Verified live: all 6 target/non-target suburbs checked now read OK; totals ERROR count dropped 12→9.
**Files:** `scripts/main_site_health_check.py`.
**Recurrence:** First. Lesson: a comparison written as `!=` when the real intent is "not stale" needs to check direction, not just equality — the same subtlety as the earlier `expected_last_run()` cadence-mismatch bugs found in this file, different mechanism (direction-blind vs wrong-baseline).

## [FIELDS-AUTOMATION-GH-ACTIONS] Investigated the 3 known-broken GitHub Actions workflows properly — 2 root causes, not 1 — 04:06 AEST
**Context:** A Task 0.5 sweep re-surfaced 3 GitHub Actions failures (Light Rail Stage 3, Market Insight, Market Shift — all flagged since `[SYSTEMS-HEALTH-CONSOLIDATION]` yesterday) that had been noted but never actually investigated this session. Rather than wave them through as "already known," pulled the actual failure logs via `gh run view --log-failed`.
**Finding — these are TWO unrelated problems wearing the same "3 broken workflows" summary:**
1. **Light Rail Stage 3** (5 straight weekly failures) and **Market Shift** (regressed the last 2 runs) both fail with `anthropic.BadRequestError: ... Your credit balance is too low to access the Anthropic API` — the exact same root cause as the already-known "Vision provider — Anthropic (primary): credit balance too low" Systems Health row. Confirms the blast radius is wider than previously understood (also breaks off-VM content-generation workflows, not just on-VM photo/floor-plan analysis) — still blocked on Will topping up credits, nothing to fix in code.
2. **Market Insight** is a GENUINELY DIFFERENT bug: its scheduled runs actually SUCCEED at generating content ("Done. Pushed: 2 | Skipped: 0 | Errors: 0") and only fail on the final `git push` — `error: failed to push some refs`. All 3 workflows run on the same self-hosted runner (per `[[fields_automation_self_hosted_runner]]`) on similar nightly/weekly schedules and all had an unguarded `git add && git commit && git push` with no pull/rebase first — a classic race when two scheduled workflows land commits on `main` close together.
**Fix:** added `git pull --rebase --autostash origin main` immediately before `git push` in all three workflow files (`market_insight.yml`, `market_shift.yml`, `light_rail_stage_3.yml` — applied to all three since Market Shift/Light Rail will hit the identical race the moment the credit issue is resolved and they start succeeding at generation again, not just to the one that already proved it). Standard, well-established pattern — not live-verified via an actual trigger (deliberately didn't force a real content-publish at 4am just to test; Market Insight's fix will validate naturally on its next Monday 7am AEST scheduled run, or whenever manually dispatched).
**Files:** `.github/workflows/market_insight.yml`, `.github/workflows/market_shift.yml`, `.github/workflows/light_rail_stage_3.yml` (all `fields-automation` repo, not `Fields_Orchestrator`).

The credit-balance cause is now confirmed in 3 separate places (Vision provider, Light Rail, Market Shift) — worth escalating to Will as one consolidated ask ("top up Anthropic API credits — currently blocking photo/floor-plan analysis AND 2 of 3 broken content-generation workflows") rather than three separate mentions.
