## [GOOGLE-INDEXING-SUBMIT-NEW-NEVER-PERSISTS-STATE] Nightly indexing submission silently full-catalog-resubmits forever, starving new listings of quota — 13:00 AEST
**Context:** Investigating Will's running-doc complaint ("still not seeing any of our listings editorial
coverage in google results... why have all 17k off-market pages been indexed almost immediately but none
of our listings editorial pages"). Checked live GSC indexation for a sample of active-listing editorial
pages old enough to judge (>1 week): 8/10 were actually "Submitted and indexed" — the blanket "none
indexed" read was mostly a timing artifact (95/110 core-suburb editorial pages were published in a single
2026-07-21/22 burst, too new to have been crawled yet). But 3 properties (38 + 83 Glen Eagles Drive Robina,
157 Christine Avenue Burleigh Waters) were genuinely stuck, and organic PostHog traffic to `/property/`
pages IS flowing (real google.com/bing.com/duckduckgo.com referrers, 14-day window) — so tracking itself
isn't broken. Investigating why NEW listings don't get the same fast indexing off-market pages get led to
the real systemic bug.
**Root cause:** `google_indexing.py cmd_submit_new()` (nightly cron `0 1 * * *`) is meant to submit only
URLs changed since its last run, tracked via `logs/indexing-last-run.json`. That state file has **never
existed on disk** — `grep -c InvalidURI logs/google-indexing.log` shows 264 historical Mongo-connection
crashes plus, independent of those, the code path itself never writes state on a first-ever run: when
`last_run` is unset, `cmd_submit_new()` falls back to `return cmd_submit_all()` — and `cmd_submit_all()`
never touches the state file. Every single night, `last_run` reads as unset, so the job re-submits the
ENTIRE property catalog (1700+ URLs) capped at the 200/day Indexing API quota, in whatever order the DB
query returns them (only loosely "static + for-sale first") — burning the day's entire quota on a mix of
old/already-indexed URLs before genuinely NEW listing pages (like 24 Elfin Street, published <24h before
Will's note) ever get a proactive submission. This is a permanent, self-perpetuating failure: it can never
transition into the efficient incremental mode because the very branch meant to bootstrap it doesn't save
the bootstrap.
**Fix:** After the `cmd_submit_all()` fallback call in `cmd_submit_new()`, write `indexing-last-run.json`
(same format as the normal incremental path). Verified via `py_compile`; did not live-run `cmd_submit_all()`
again since today's 200/day quota is already exhausted from earlier manual resubmit attempts — tonight's
1am cron run will exercise the fix for real (state file should exist afterward, and tomorrow's run should
use the fast incremental path querying only `last_updated >= last_run`, well under 200/day for 3 suburbs).
**Files:** `scripts/google_indexing.py` (`Fields_Orchestrator`, 1 commit).
**Recurrence:** First for this specific bug. Worth a Systems Health follow-up: add a check that
`indexing-last-run.json` exists and is recent, so a regression here is caught automatically instead of
requiring someone to notice indexation is silently starved.

## [OFFMARKET-SCHEMA-VERSION-STAMP] Built the schema-version invalidation fix — closes the recurring stale-cache bug class — 14:49 AEST
**Context:** The `offmarket_intel`/`offmarket_positioning` caches (cached forever, by design, per-property) had bitten twice in one session with the same failure mode: a backend logic change shipped, but already-cached docs kept silently serving pre-fix content until someone happened to notice and manually reset the doc in Mongo. **Fix:** Added `INTEL_SCHEMA_VERSION`/`POSITIONING_SCHEMA_VERSION` constants to `offmarket_intel_poller.py` (both currently `1`), stamped onto every write to both collections — success AND error paths, so a permanently-failing property doesn't get retried on every single pageview once the version starts being tracked. Mirrored `CURRENT_SCHEMA_VERSION` constants in both Netlify endpoints (`offmarket-intel.mjs`, `offmarket-positioning.mjs`); a cached doc whose `schema_version` doesn't match is **force-invalidated** — the endpoint responds `{"status":"pending"}` and re-queues, never serving the mismatched content even transiently (deliberately stricter than the existing age-based `FRESH_DAYS` staleness check, which still soft-serves old content while a background refresh runs — a version mismatch can mean the cached content is outright WRONG, not just old, so nothing gets served until the fresh recompute lands). No shared constant between Python and JS in this codebase, so keeping the two numbers in sync is a documented manual-discipline contract (comment in both files pointing at the other), not enforced — the honest limitation of this fix.
**Verified end-to-end, live, with zero manual Mongo intervention** (the actual point of building this): confirmed 5 Glenferrie Drive's real cached doc had no `schema_version` field; hit the live `/api/v1/offmarket-positioning` endpoint exactly as a real pageview would — it correctly responded `{"status":"pending"}` (did NOT serve the stale content) and re-queued the doc in Mongo; waited for the poller's normal cycle to recompute and stamp `schema_version:1`; re-hit the endpoint and confirmed it now serves the fresh, correct result (`school_walk_family`, not the old `value_liquidity_play` bug).
