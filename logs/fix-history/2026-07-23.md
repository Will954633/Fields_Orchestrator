## [STEP6-OVERPASS-406-SILENT-ZERO-VALUATIONS] Nightly valuation step 6 had been valuing 0 properties — Overpass now blocks the default python-requests User-Agent — 16:05 AEST
**Symptom:** OPS_STATUS showed step 6 "Batch Valuate Properties" failed. `system_monitor.process_runs` showed it failing on 2026-07-20, 07-21 and 07-22 — three consecutive nights, always in ~61s, always `properties_valued: 0, properties_failed: 34`. Step logs showed every one of the 34 properties hitting `OSM enrichment failed` → `✗ Enrichment incomplete - skipping valuation`, with `HTTPConnectionPool(host='overpass-api.de', port=80) ... [Errno 101] Network is unreachable`.
**Two misleading signals worth recording, because both would have sent the next person the wrong way:**
1. **The orchestrator reported the step as SUCCESS.** `run_summary.json` for the run says `steps_failed: 0` and the local step dir is `14_step_6_property_valuation_model` with a clean result. The script exits 0 even when it values nothing; only the script's own `process_runs` record (written under its internal name "Batch Valuate Properties") carries the real 0/34 outcome. A step can be green at the orchestrator level and totally ineffective at the business level.
2. **`Errno 101 Network is unreachable` was NOT the cause.** This VM genuinely has no global IPv6 address and no IPv6 default route (confirmed: `ip -6 addr show scope global` and `ip -6 route show default` both empty, `curl -6` fails), so the IPv6 leg of `overpass-api.de` does fail that way — but urllib3 tries every resolved address and only raises the LAST error, so the IPv6 failure was merely the tail of the list. The real rejection happened on IPv4 first. Chasing the IPv6 message would have produced a networking fix for a non-problem.
**Root cause (isolated by controlled experiment, not inference):** `overpass-api.de` now rejects the default `python-requests` User-Agent. Four variants run against the live endpoint with the same query: (A) raw-body POST + default UA — the exact call the script made — **406**; (B) form-encoded + default UA — **406**; (C) raw-body + identifying UA — **504**; (D) form-encoded `data={"data": query}` + identifying UA — **200, 100,726 bytes of real OSM data.** Only D works. The script had been doing A.
**Business impact (this is why it mattered more than a red row):** `iteration_08_valuation` is consumed on the live site by `AnalysePropertyPage` (the Analyse Your Home flow) and `netlify/functions/valuation.mjs` — i.e. it sits on the seller-conversion path, not in a back room. Coverage on the three target suburbs was **36 of 224 for-sale properties (16%)**, and the most recent successful write was Robina 2026-07-15, **Burleigh Waters 2026-04-10, Varsity Lakes 2026-04-08** — over three months stale.
**Fix:** `osm_feature_definitions.py` — (1) default endpoint moved to HTTPS; (2) POST is now form-encoded with an identifying `User-Agent` (`FieldsEstate-PropertyValuation/1.0` + contact URL/email, which is also what the OSM usage policy asks for); (3) `rate_limited_request()` gained bounded exponential-backoff retry for transient `500/502/503/504` and for timeouts (4 attempts, 5→10→20s capped at 60s), because a 504 previously returned `None` and that hard-failed the whole property's valuation. The 429 branch is now also bounded instead of recursing indefinitely.
**Verified (by running it, not by reading it):** OSM extractor returns real data for all three target suburbs — Robina (nearest road `footway` 47.0 m, water 16.4 m), Burleigh Waters (`residential` 41.5 m, water 11.6 m — correctly waterfront), Varsity Lakes (`tertiary` 9.1 m). Then the actual production script end-to-end: `batch_valuate_with_tracking.py --limit 3` → **3 queried, 3 valued, 0 failed, 3 dual DB inserts** (was 0/34), and the retry logic visibly fired mid-run (`API error: 504, retrying in 5s (attempt 1/4)`) and recovered. Confirmed the writes landed in Mongo: 20 Chantilly Place $1,445,218; 17 Springvale Street $1,427,504; 41 Camberwell Circuit $1,392,472.
**Checked for siblings before claiming a pattern:** only two files in the estate call Overpass. `scripts/property_reports/walking_distances.py` was read in full and is already correct (HTTPS, form-encoded, real User-Agent, three mirrors, backoff) — so this was a single-script bug, not "the same bug in N scripts." Per the self-audit rule, the full surrounding context was read rather than acting on a grep line.
**Files:** `Property_Valuation/04_Production_Valuation/osm_feature_definitions.py` (`Will954633/Property_Valuation`, commit `f2e7b18`). Revert = revert that commit.
**Recurrence:** The 406/User-Agent block is **First**. The underlying "Overpass transient failure with no retry" class is the **2nd** instance — `[MINI-SITE-DAY6]` (2026-03) hardened `walking_distances.py` with mirrors + backoff for exactly this, and that hardening was never back-ported to the valuation path. ## [OFFMARKET-DRIVERS-CHERRYPICK] Drivers card silently dropped its only bearish indicator — caught on live verification — 19:34 AEST
**Symptom:** After the macro-drivers rebuild published, the card listed Wage Growth ↑, Consumer Spending ↑, Dwelling Supply ↓(bullish), Inflation ↓ — all four positive — and asserted *"Of the 4 indicators here, 4 are leading measures... pointing the same way as each other."* Housing Lending, the single bearish reading, was gone.
**Root cause:** I ranked the lagging indicator LAST (rank 9) so it would sit beside its "lags by 3.5 months" caveat, then applied `.slice(0, 4)` to the ranked list. With five-plus signals available, the slice cut exactly the one I'd deliberately ranked last. The result wasn't just a missing row — it was **cherry-picking**: a card showing only favourable indicators, claiming unanimity, with a lead/lag synthesis that had no exception left to explain. Precisely the failure the editorial rules exist to prevent, produced by an ordering change meant to improve honesty.
**Fix:** the lagging indicator is now partitioned out BEFORE the top-N cut and appended after it (`leadingOnly.slice(0, lagging.length ? 3 : 4)` + the lagging one), so it can never be truncated away. Synthesis copy also split: the "N of M are leading, X is the exception" form only renders when a lagging indicator is actually present; otherwise it uses an "all N are leading measures" phrasing rather than the clunky "of the 4 indicators here, 4 are leading".
**Verified live:** card now reads Wage Growth / Consumer Spending / Dwelling Supply / **Housing Lending (tagged `lagging`, with its caveat inline)**, closing on *"Of the 4 indicators here, 3 are leading measures... Housing Lending is the exception: it lags prices by around 3.5 months, so it describes where the market has been rather than where it is now."*
**Lesson:** a top-N truncation applied after a deliberate re-ordering can silently delete the very item the ordering was designed to position — and when the deleted item is the only counterweight, a presentation bug becomes an honesty bug. Worth checking any `slice()` that follows a `sort()` for what it's cutting, not just what it's keeping.

## [OFFMARKET-POSITIONING-KICKOFF] The slowest task in the flow wasn't starting until the first swipe — 19:49 AEST
