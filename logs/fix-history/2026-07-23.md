## [QUARTER-COMPLETENESS-FALSE-POSITIVE] Systems Health flagged a working fix as broken — direction-blind comparison — 03:35 AEST
**Context:** A loop cycle's Task 0.5 sweep found 3 new ERROR rows ("Indexed prices — quarter completeness", Burleigh Waters/Robina/Varsity Lakes) that hadn't appeared in earlier cycles' checks that same session.
**Symptom:** `raw sold data has 52 sales in Q1 2026 but the precomputed series' latest complete quarter is Q2 2026` — read at a glance like a regression.
**Root cause:** the check (added earlier today per `[SYSTEMS-HEALTH-CONSOLIDATION]`/`[PROPERTY-TIMELINE-REFRESH-DEAD]`, see `2026-07-22.md`) computes an `expected_label` two quarters back from now (a grace window, since `property_timeline` is weekly-refreshed and `indexed_prices` monthly-precomputed) and flags ERROR whenever `latest_complete_period != expected_label`. Plain inequality doesn't distinguish "behind schedule" from "ahead of schedule" — and this morning's `[PROPERTY-TIMELINE-REFRESH-DEAD]` fix had, by this evening, genuinely caught the precomputed series up to Q2 2026 — ONE quarter ahead of the check's own 2-quarter grace expectation. The check flagged the exact outcome the morning's fix was meant to produce as an error.
**Fix:** `main_site_health_check.py`'s quarter-completeness check now parses both periods into `(year, quarter)` tuples and only flags ERROR when the precomputed series is genuinely BEHIND the expected quarter (`<`), never merely not-equal. Verified live: all 6 target/non-target suburbs checked now read OK; totals ERROR count dropped 12→9.
**Files:** `scripts/main_site_health_check.py`.
**Recurrence:** First. Lesson: a comparison written as `!=` when the real intent is "not stale" needs to check direction, not just equality — the same subtlety as the earlier `expected_last_run()` cadence-mismatch bugs found in this file, different mechanism (direction-blind vs wrong-baseline).

## [ENV-LOADING-STANDARDIZED] Migrated all 14 ad-hoc .env parsers onto python-dotenv — self-audit point 8 — 08:08 AEST
**Context:** Will asked what to do about the self-audit's tooling-gap findings. This is the concrete fix for the `.env`-loading friction (the second half of point 8) — 14 scripts each hand-rolled their own `.env` parser instead of using the already-installed `python-dotenv` library, which is exactly the class of bug that caused `[GOOGLE-INDEXING-INVALIDURI]` (2026-07-22) in the first place.
**Fix:** replaced each script's `load_env()`/`set_env_from_file()`/`load_env_file()` body with `load_dotenv(<path>, override=False)` — `override=False` (python-dotenv's default) preserves the exact same semantics as the old `os.environ.setdefault(...)` pattern (an already-exported var always wins), so this is a pure implementation swap, not a behaviour change. Kept every function's name/signature/call sites untouched — only bodies changed, so nothing calling these functions needed updating.
**Verified:** all 14 files compile (`py_compile`); functional smoke test on 2 of them (`google_indexing.py`, `main_site_health_to_sheet.py`) confirms `COSMOS_CONNECTION_STRING` loads clean (no stray quote characters) with zero pre-exported environment, the exact scenario that broke before.
