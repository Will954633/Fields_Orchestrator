## [CEO-STEP116-RETRY-V2] Fix step 116 retry regression — _id cursor instead of skip() — 08:45 AEST

**Symptom:** Step 116 failed again on 2026-03-22 run despite yesterday's retry fix. Log showed 8 consecutive 429 retries stuck at 80 docs on burleigh_heads.
**Root cause:** Yesterday's fix used `.skip(len(results))` on retry, but Cosmos DB's `.skip()` on an unindexed collection with `$or` query re-scans from offset 0 to the skip point — consuming MORE RUs than the original query. Each retry made the 429 worse.
**Fix:** Replaced skip-based retry with `_id > last_seen_id` cursor approach. Tracks last successfully read `_id` and resumes from there on 429. Also reduced batch_size from 20 to 10, added 300ms pause every batch_size docs, increased max retries from 8 to 10, and increased inter-suburb cooldown from 2s to 5s.
**Files:** `scripts/data_quality_validator.py`
**Recurrence:** Third occurrence (2026-03-21 22:45, 2026-03-22 02:45, 2026-03-22 20:55).
