## [MINISITE-COMPSET-WHY] Per-home "why this rank" explanation line — 17:40 AEST
**Symptom:** Not a bug — feature build (Will's request). The ranked-comparison panel showed each home's match % + a feature list, but it wasn't immediately obvious WHY a home sat at #4 vs #5 without expanding the breakdown.
**Root cause/insight:** Rank is fully determined by the weighted score, and we already persist each factor's `contribution` (= weight × normalisedDistance), which sums to the score. So "why here" needs no new model or data — it's a deterministic readout: strengths = factors that essentially matched (distance ≤ 0.12), drags = the 1-2 factors with the highest contribution (their biggest push down the list). Reading two adjacent homes' lines makes the gap self-evident.
**Fix:** Frontend-only. Added `whyRanked()` + `strengthPhrase()`/`dragPhrase()` helpers to RankedComparison.tsx that turn `breakdown[]` into a one-line "Matches on X, Y. Ranked here: <real-value gap>." subtitle, rendered always-visible on every ranked row (strengths in grass, drags in copper). Drag phrases quote the actual gap (e.g. "22% smaller block", "4.8 km away", "1 bedroom fewer", "no pool", "listed 10% above your guide") — factual, no advice/superlatives (editorial-safe). Two thresholds (strength ≤0.12, drag ≥0.30) with a neutral middle band so a factor is never both. No backend change, no re-resolve — data already on the doc.
**Verification:** Replicated the helper against the live persisted breakdown for 25 Huntingdale — lines distinguish adjacent ranks cleanly (e.g. #3 77% "4.0km away, adds shed" vs #4 77% which matches on price guide but is "4.8km away, 1 car space fewer"). tsc --noEmit clean, npm run build clean.

## [STEP105-DICT-IMAGES] Photo analysis silently failed every property → cascaded floor-plan gap — 18:05 AEST
**Symptom:** Will flagged that listed properties with floor-plan images had no floor-plan *data*. Investigation of the core-suburb gap (38 active listings lacking `floor_plan_analysis`, 108 lacking step-106 `ollama_floor_plan_analysis`) traced upstream to step 105 (photo analysis): the last 3 nightly runs logged "Successful: 0, Failed: 177" — "No images found" for every property — yet the step reported exit 0 / success:true.
**Root cause:** Two compounding bugs:
1. The scraper switched `scraped_data.images` to a list of dicts (`{"url": ...}`). `PropertyWorkerMulti._extract_images` (and `worker.py`) only kept `isinstance(item, str)` entries, so dict-shaped payloads yielded an empty list. Worse, the `if/elif` chain selected `scraped_data.images` whenever the key merely *existed* and never fell through to the populated `property_images` (48–60 string URLs were sitting right there). Result: "No images found" → no `ollama_image_analysis` → step 106's candidate query (gated on `ollama_image_analysis` existing) never picked the property up → no floor-plan analysis. 88 core for-sale listings blocked (33 Robina / 19 VL / 36 Burleigh).
2. `run_production.py` only exited non-zero on a fatal exception. An all-failed batch (graceful `return False` per property) completed normally with exit 0, so the orchestrator marked step 105 successful and the failure stayed invisible for months.
**Fix:** (1) Rewrote `_extract_images` in `worker_multi.py` + `worker.py` to coerce dict entries (`url`/`src`/`image_url`) to URL strings and to accumulate across all sources with real fallback instead of elif-stopping. Mirrors step 106's existing `_coerce_url_list` (which is why floor-plan analysis worked but photo analysis didn't). (2) Added a guard in `run_production.py`: if `total_failed > 0 and total_successful == 0`, log + `sys.exit(1)` so a wholesale failure surfaces as a failed step. Verified the new extractor yields images for all 94 previously-blocked core docs (was 0).
