## [STEP113-RUNTIME] detect_withdrawn grinding 6h on flaky BrightData — 20:40 AEST

**Symptom:** Nightly step 113 ran 21,720s (~6h) then was killed (`Process timed out`), logging `could not determine status` for every for_sale listing. Recurring — watchdog has flagged step 113 repeatedly.
**Root cause:** Not a credential break — BrightData Web Unlocker creds/zone tested healthy (live 200 + 525KB Domain HTML). But the Web Unlocker is intrinsically flaky/slow for Domain (module's own notes: "~30% min_size flakiness", "cold-start can exceed 90s"). On a degraded BD night, requests retry-and-timeout (up to 5 retries × 60s × ~150 listings) with no global ceiling, so the step grinds until the orchestrator's ~6h process timeout — monopolising the pipeline window. The step has `--no-fail` so per-request errors don't fail it, but the wall-clock blowout does.
**Fix:** Added two bounds to `run_withdrawn_detection`: (1) global wall-clock budget `MAX_RUNTIME_MIN` (default 40, env `WITHDRAWN_MAX_RUNTIME_MIN`) checked before each listing; (2) circuit breaker `CIRCUIT_BREAKER_ERRORS` (default 15, env `WITHDRAWN_CIRCUIT_BREAKER`) that aborts after N consecutive fetch errors (BD clearly degraded). Both exit gracefully, leaving unchecked listings for next run. Verified: with breaker=3 + mocked errors, aborts after exactly 3 instead of all 72+ listings. Tonight's live run (currently on step 101) will pick up the new code when it reaches 113.
**Files:** scripts/detect_withdrawn.py (pushed to GitHub)
**Recurrence:** Nth — step 113 timeouts/failures are a recurring watchdog flag; this is the first time the 6h-from-BD-flakiness runtime was bounded.
