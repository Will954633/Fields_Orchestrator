# Fix History — 2026-05-29

## [BLOB-BACKUP-AUTH] GCS rsync silently failed under sudo — 09:25 AEST
**Symptom:** Initial 994k-file sync of `/data/blobs/` → `gs://fields-blob-backup` finished with 0 bytes uploaded. Every file logged `AccessDeniedException: 403 Provided scope(s) are not authorized`. Discovered when verifying backup completeness ahead of Azure deletion.
**Root cause:** Sync ran via `sudo gsutil`. Under sudo, gsutil uses root's gcloud context → the VM's default GCE service account (`419034603899-compute@developer.gserviceaccount.com`), which has only `devstorage.read_only` scope. Cannot write to GCS. Without sudo, gsutil uses `will.simpson@blueoceans.com.au` user creds, which have write permission.
**Fix:**
- Killed failing root sync (PID 2288403) and the cron-spawned variant.
- Re-ran `gsutil -m rsync -r /data/blobs/ gs://fields-blob-backup/` WITHOUT sudo (no sudo needed — `/data/blobs/` is owned by `projects:projects` 755).
- Updated cron to drop `sudo`. Cron now lives in user `projects`'s crontab.
- Documented gotcha in `00_Run_Commands/operations/backups.md` and in memory file `gcs_blob_backup.md`.
**Files:** user crontab, `00_Run_Commands/operations/backups.md` (new), memory `gcs_blob_backup.md` (new).
**Recurrence:** First. Likely to bite again if someone wraps a future GCS operation in sudo out of habit.
