## [APPR-V4-SPLICE-01] §02R duplicated in V4 renders — missing splice divider — 09:21 AEST

**Symptom:** Every V4 appraisal rendered "Three buyers. One outbids the field." twice — once without willingness-to-pay rows (the new template-spliced version), once with WTP rows (the old hardcoded version). Visible in the 5 Pinnacle Court PDF at pages 7 and 8.

**Root cause:** `generate_appraisal_v4.py:splice()` finds the start/end of a section block by locating the named PAGE marker, then walking *up* to the nearest `<!-- ==== -->` divider. Every section boundary in `preview.html` has this divider directly above its PAGE comment — except `<!-- PAGE 06 — SPREAD 03 LEFT — VALUATION (thesis) -->` at line 2195, where the divider was missing. `rfind("<!-- ====", 0, end_idx)` then fell back to the divider *inside* the §02R block (the one between the PAGE 07 comment header and `<div class="page">`), so the splice preserved the entire old §02R block as the "above_end" tail.

**Fix:** Inserted the missing `<!-- ============================================================ -->` line directly above the PAGE 06 SPREAD 03 LEFT comment in `preview.html`. Regenerated 5 Pinnacle Court and confirmed the §02R headline now appears exactly once.

**Files:** `09_Appraisals/Version_Four/preview.html`.

**Verification:** `grep -c "Three buyers" artifacts/appraisals_v4/690bd7e68b8f5465926044f2_20260516T092106.html` → 2 (one comment + one headline). PDF page 6 = §02L thesis, page 7 = §02R personas, page 8 = §03L thesis — clean ordering.

**Recurrence:** First.

## [APPR-V4-NAME-01] "for Dee" leaked through to page 19 sign-off — 09:31 AEST

**Symptom:** Page 19 closing sign-off read "This report was built for 5 Pinnacle Court and for Dee." for every V4 render — the original analyst name from the 13TC template was not being replaced.

**Root cause:** Post-process substitution in `generate_appraisal_v4.py` only replaced `Prepared for Dee` (cover band) and the inside-cover `<span class="name">Dee</span>`. The standalone "for Dee" embedded in the page-19 closing paragraph was not in the substitution list.

**Fix:** Added two branches: if `prepared_for_name == "the Owner"` (the fallback used when no pipeline record exists), strip " and for Dee" entirely so the sentence reads as a single-subject statement. Otherwise replace "for Dee" with f"for {prepared_for_name}".

**Files:** `scripts/generate_appraisal_v4.py`.

**Verification:** Re-rendered 5 Pinnacle Court — page 19 reads "This report was built for 5 Pinnacle Court. The strategy in it is genuine — not a template…".

**Recurrence:** First.

## [APPR-V4-PENDING-01] §03R / §03 receipts / recommendation pages render "Analyst review required" state when valuation absent — 09:30 AEST

**Symptom:** Subjects without a populated `valuation_data` (e.g. sold-cohort properties used for template testing, or bridged appraisals before the analyst has confirmed comparables) produced visibly broken V4 pages: "—. The range, derived." (page 9), an empty card area below the backtest stat (page 10), and "$0 – $0" / "—" boxes on the recommendation pages (11 + 18).

**Root cause:** The renderers blindly formatted empty values. `_format_dollar_compact(None)` returns `""` which yielded `" – ."` as the §03R headline range; `len(cards) == 0` collapsed §03 receipts to header + backtest with nothing in between; the recommendation Jinja templates had `{% else %}—{% endif %}` fallbacks that printed bare em-dashes in the big price boxes.

**Fix:** Added a `pending_review` flag to each of the three `data_pull` builders, derived from the actual data state:
- `section_03_right`: `pending_review = not (range_low and range_high)`
- `section_03_receipts`: `pending_review = len(cards) == 0`
- `section_recommendation`: `pending_review = not (listing_price and target_low)`

Each renderer now branches on this flag and emits a clean "ANALYST REVIEW REQUIRED" notice block with copper-bordered card, monospace eyebrow, plain-language description of what populates the page, and an explicit workflow line ("Ops dashboard → Appraisal Pipeline → confirm comparables → re-render report"). The §03R page still shows the cohort-median anchor block (which is computed from catchment queries and remains meaningful even before the engine runs) and FA-03 below. The §03 receipts page still shows the 11.4% backtest stat. The recommendation pages drop the placeholder $0 boxes entirely.

**Files:** `scripts/appraisal_template/render.py`, `scripts/appraisal_template/data_pull.py`.

**Verification:** Re-rendered 5 Pinnacle Court (no valuation_data, no pipeline record) → pages 9, 10, 11, 18 all show the clean pending-review state; pages 1–8, 12–17, 19 unaffected. The same renderer will produce the full content on a bridged subject once `valuation_data.confidence.range` and `pipeline.recommendation` are populated via the ops dashboard.

**Recurrence:** First.
