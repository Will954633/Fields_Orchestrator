## Open decisions for you

### 1. Bright Data spend
The 50-property test used 50 requests. I couldn't programmatically determine per-URL cost. Rough estimates:
- At `$0.005/URL` → full pass = **$140**
- At `$0.025/URL` → full pass = **$700**
- Bandwidth ~10 GB total at typical $10/GB residential = **$100**

**Check Bright Data dashboard for actual usage from today.** If the test/early run is much costlier than expected, kill the process.

### 2. Throttle rate
You specified 1 req/2s = 0.5/sec dispatch. Observed actual rate is ~0.27/sec (Bright Data Web Unlocker latency dominates). Doubling to `--rate 1` and bumping workers to 60 would cut ETA from 30h to ~15h. Probably safe — current volume is moderate vs your existing target-market scrapes.

### 3. Image durability
v2 scrape stores Domain CDN URLs, not binary files. Same fragility as `property_images_original` — Domain rotates URLs when a listing is withdrawn. For true permanence, follow-on work: extend the scraper to download each hero image to `/home/fields/property_images/<id>/00.jpg` + serve via a Netlify proxy. Defer until current pass is done.

### 4. Off-market view
v2-scraped data sits on cadastral docs (no `listing_status`). Existing user-facing pages only show active listings. To expose the 28k off-market valuations + sale histories you'd need either:
- A new `/property-profile/:slug` route on the website
- Or fold these into market metrics / comparable analysis tools you already have

Not blocking. Worth a roadmap conversation.
